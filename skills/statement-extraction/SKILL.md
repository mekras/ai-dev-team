# Навык: извлечение утверждений из данных

## Назначение

Этот навык задаёт порядок извлечения из первичных данных и их нормализованных
представлений проверяемых утверждений, пригодных для последующего
проектирования.

Навык нужен для того, чтобы переход от данных к содержательным выводам
происходил последовательно, с сохранением связи с источником и с явным
разделением фактов, наблюдений и выводов.

`statement-extraction` — канонический навык извлечения проверяемых единиц
корпуса. Он покрывает и конечные утверждения, на которые могут ссылаться
документы проекта, и промежуточные кандидаты для последующего
`source-impact-audit`.

## Когда применять навык

Навык применяется, если нужно:

- выделить из материала значимые утверждения;
- подготовить основу для требований, правил, решений или документации;
- зафиксировать содержательные единицы, которые можно проверять и сопоставлять;
- отделить подтверждаемое источником содержание от интерпретации.

Если пользователь явно просит заполнить или проверить промежуточную таблицу
кандидатов для `source-impact-audit`, применяй этот навык, но используй
локальные ID промежуточного артефакта, а не конечный ссылочный формат
`<SOURCE-ID>-NNN`.

## Входные данные

Для применения навыка должны быть доступны:

- первичный материал или его нормализованное представление;
- сведения о происхождении источника;
- известные ограничения источника;
- понимание, для какого следующего шага извлекаются утверждения.

Если этого недостаточно, ограничение должно быть зафиксировано явно.

## Порядок применения

### 1. Определить цель извлечения

Нужно зафиксировать:

- из какого материала извлекаются утверждения;
- для какого следующего шага это делается;
- какие типы утверждений нужны;
- какие выводы ещё нельзя делать на этом этапе.

### 2. Проверить основание

До извлечения должно быть понятно:

- на каком материале строится работа;
- является ли материал первичным или нормализованным;
- какие ограничения этого материала уже известны;
- можно ли проследить каждое утверждение до основания.

Если основание неясно, извлечение выполнено с нарушением навыка.

### 3. Выделить утверждения

Нужно извлекать утверждения так, чтобы каждое из них:

- относилось к конкретному фрагменту материала;
- было сформулировано проверяемо;
- не смешивало несколько разных смыслов без необходимости;
- не включало лишнюю интерпретацию.

### 4. Определить статус утверждения

Для каждого утверждения нужно по возможности понять:

- это факт;
- это наблюдение;
- это вывод;
- это спорный или неполный фрагмент.

Если статус нельзя определить уверенно, это должно быть явно отмечено.

### 5. Назначить идентификатор утверждения

Для каждого утверждения нужно назначать устойчивый идентификатор, пригодный для
ссылок из анализа, отчётов о влиянии и производных документов.

До нумерации утверждений должен быть определён короткий идентификатор источника.

Требования к идентификатору источника:

- не более 5 символов;
- только прописные латинские буквы;
- один и тот же идентификатор во всех слоях корпуса.

Для конечных утверждений, на которые должны ссылаться документы проекта,
используй формат:

- `<SOURCE-ID>-001`
- `<SOURCE-ID>-002`

где:

- `<SOURCE-ID>` берётся из идентификатора источника;
- числовая часть остаётся уникальной внутри одного источника.

Примеры:

- `HWD-001`
- `WTDG-014`
- `ASKLA-006`

Идентификатор должен:

- быть уникальным в пределах всего корпуса;
- не меняться без необходимости после появления в Git;
- использоваться одинаково в самом файле утверждений и во всех ссылках на него.

Для промежуточных артефактов, на которые не должны ссылаться документы проекта,
допустим отдельный локальный формат. Такой ID должен явно отличаться от
конечного формата и не должен использоваться как постоянная ссылка в
производной документации.

Не используй локальные идентификаторы вида `S-001` или `001` для конечных
утверждений.

### 6. Зафиксировать ограничения

Нужно отдельно отмечать:

- неполное основание;
- неоднозначную формулировку;
- зависимость от качества нормализации;
- сомнительную интерпретацию;
- необходимость проверки человеком.

### 7. Отделить извлечение от следующего этапа

Извлечение утверждений не должно включать:

- формулирование требований;
- выбор проектного решения;
- введение нового правила;
- объединение утверждений в окончательный документ;
- утверждение спорного вывода как принятого.

Если это уже требуется, работа должна переходить к следующей роли или следующему
навыку.

### 8. Зафиксировать результат

По итогам должно быть понятно:

- какие утверждения извлечены;
- каков их статус;
- на каком основании они извлечены;
- какие ограничения нужно учитывать;
- для какого следующего шага этот набор пригоден.

## Результат применения навыка

Результатом должен быть набор утверждений, который:

- связан с конкретным материалом;
- разделён по статусам;
- не подменяет данные проектными выводами;
- пригоден для следующего этапа обработки.

## Что считается ошибкой применения навыка

- утверждение нельзя проследить до источника;
- вывод выдан как факт;
- спорный фрагмент подан как однозначный;
- извлечение незаметно перешло в требования или решения;
- ограничения источника не отражены в результате;
- утверждения настолько расплывчаты, что их нельзя проверить.

## Связь с другими элементами продукта

Этот навык должен применяться вместе:

- с принципом корпуса первичных данных;
- с навыком приёма первичных данных;
- с навыком нормализации первичных данных;
- с ролью `source-inventory`;
- с ролью нормализатора данных;
- с ролью извлекателя утверждений;
- с контрольным слоем, если результат извлечения влияет на значимые решения
  проекта.
