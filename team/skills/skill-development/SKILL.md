# Навык: разработка навыков агента

## Назначение

Этот навык задаёт порядок создания, изменения и сопровождения навыков агента.

Навык нужен для того, чтобы новые навыки появлялись не как обычные справочные
тексты, а как проверяемые рабочие элементы продукта: с понятным основанием,
точной областью применения, проверкой срабатывания и оценкой пользы.

## Внешние основания навыка

Навык опирается на два разделённых источника корпуса:

- `ASKL`:
- первичный источник: `knowledge/primary/ASKL/source.md`;
- нормализованный слой: `knowledge/normalized/ASKL/source.md`;
- извлечённые утверждения: `knowledge/statements/ASKL.md`;
- `PPRX`:
- первичный источник: `knowledge/primary/PPRX/source.md`;
- нормализованный слой: `knowledge/normalized/PPRX/source.md`;
- извлечённые утверждения: `knowledge/statements/PPRX.md`.

Положения источника не являются правилами проекта сами по себе. Они используются
как основание для проектного навыка после анализа применимости к этому продукту.

## Когда применять навык

Навык применяется, если нужно:

- создать новый навык агента;
- существенно изменить существующий навык;
- изменить описание, по которому навык должен срабатывать;
- добавить справочные файлы, шаблоны, скрипты или другие вспомогательные
  материалы навыка;
- проверить, нужен ли навык как отдельный элемент продукта;
- разобрать сбой применения навыка и решить, как его сопровождать.

Навык не применяется для обычной правки документации, если она не меняет
назначение, область применения, срабатывание или порядок работы навыка.

## Входные данные

Для применения навыка должны быть доступны настолько, насколько это возможно:

- практическая задача или повторяемая проблема, которую должен закрыть навык;
- примеры реальных запросов, задач, ошибок или артефактов;
- проектные правила и ограничения;
- предполагаемая область применения навыка;
- соседние навыки, с которыми возможны пересечения;
- критерии проверки результата.

Если реальных задач, ошибок или артефактов нет, навык нельзя считать достаточно
обоснованным для включения в переносимое ядро.

## Порядок применения

### 1. Проверить необходимость навыка

Перед созданием навыка нужно ответить:

- какую повторяемую проблему он решает;
- почему обычных правил, документации или одного уточнения в задаче
  недостаточно;
- что агент будет делать неправильно, нестабильно или слишком дорого без навыка;
- почему это относится к переносимому решению продукта, а не только к локальному
  проекту;
- как будет проверяться польза навыка.

Если ответы неясны, нужно сначала собрать данные или оставить идею как кандидат,
а не создавать навык.

### 2. Определить основание

Навык должен опираться на проверяемое основание:

- реальную выполненную задачу;
- повторяющийся сбой агента;
- проектные артефакты;
- извлечённые утверждения из корпуса первичных данных;
- подтверждённое решение человека.

Нельзя создавать навык только из общего представления о том, «как правильно».
Если основание слабое, это должно быть явно отражено в статусе результата.

### 3. Спроектировать область применения

Нужно определить:

- когда навык должен применяться;
- когда он не должен применяться;
- какие соседние навыки могут пересекаться с ним;
- какие действия навык разрешает агенту делать самостоятельно;
- какие действия требуют решения человека.

Область применения должна быть достаточно узкой для точного срабатывания, но
достаточно цельной, чтобы навык не приходилось собирать из множества мелких
фрагментов.

### 4. Написать описание срабатывания

Описание навыка должно помогать агенту решить, загружать ли навык.

Хорошее описание:

- начинается с условия применения;
- описывает намерение пользователя или тип задачи;
- упоминает важные косвенные признаки задачи;
- отделяет навык от соседних случаев;
- не пересказывает внутреннюю структуру навыка.

После изменения описания нужно проверить положительные и отрицательные примеры
срабатывания.

### 5. Отобрать содержимое навыка

В основной файл навыка нужно включать только то, что агент с заметной
вероятностью сделает неправильно без этой инструкции:

- проектные соглашения;
- предметные процедуры;
- неочевидные ограничения;
- частые ошибки;
- устойчивые предпочтения проекта;
- конкретные инструменты или форматы, если их выбор важен.

Не нужно включать сведения, которые агент обычно знает без навыка, если они не
получают особый смысл в этом проекте.

### 6. Использовать постепенное раскрытие

Если навык становится большим, нужно отделять ядро от подробностей.

В основном файле остаются:

- назначение;
- условия применения;
- обязательный порядок;
- границы самостоятельных действий;
- критерии готовности.

Подробности выносятся во вспомогательные файлы, например:

- `references/` для длинных справок;
- `scripts/` для повторяемой детерминированной обработки;
- `assets/` для шаблонов и примеров.

Для каждого вспомогательного файла должно быть понятно, когда агент должен его
читать или использовать.

Для инструкций, которые влияют на поведение модели (в том числе AGENTS-подобных
точек входа), в основном файле должен быть явный минимальный контракт:

- что является обязательным для старта задачи;
- что относится к расширению и может быть вынесено в optional-разделы;
- как определить, что документ разросся и требует рефакторинга.

### 7. Проверить навык

Проверка навыка должна включать:

- реалистичные запросы, при которых навык должен срабатывать;
- близкие отрицательные запросы, при которых навык не должен срабатывать;
- ожидаемый результат для основных сценариев;
- при необходимости входные файлы;
- проверяемые утверждения о качестве результата.

Для оформления проверочных случаев используй
`skills/skill-development/assets/skill-evaluation-case-template.md`.

Для существенного навыка полезно сравнивать выполнение:

- без навыка;
- с новым навыком;
- с предыдущей версией навыка, если она есть.

Для фиксации итогов проверки используй
`skills/skill-development/assets/skill-evaluation-report-template.md`.

Если проверка не проводилась, результат должен считаться черновиком.

### 8. Сопровождать навык по фактическим сбоям

Навык нужно уточнять по реальным случаям применения.

Особенно полезны:

- случаи, когда навык не сработал, хотя должен был;
- случаи, когда навык сработал не к месту;
- ошибки агента при выполнении навыка;
- повторяющиеся вопросы человека;
- лишние инструкции, которые не влияют на результат.

Неочевидные ошибки можно фиксировать как отдельные предупреждения или
отрицательные примеры, но такой раздел не должен превращаться в склад случайных
запретов.

### 9. Зафиксировать результат

По итогам разработки или изменения навыка должно быть понятно:

- какая проблема закрывается;
- на каком основании навык создан или изменён;
- какие сценарии он покрывает;
- какие соседние случаи исключены;
- как он проверен;
- какие ограничения и открытые вопросы остаются.

## Что считается ошибкой применения навыка

- навык создан без повторяемой проблемы;
- описание навыка пересказывает содержание, но не помогает срабатыванию;
- навык дублирует очевидные знания агента;
- навык смешивает переносимые правила и локальные особенности проекта;
- большой справочный материал загружен в основной файл без условия применения;
- изменение описания не проверено на положительных и отрицательных запросах;
- единичный сбой превращён в общий запрет без анализа причины;
- навык принят как готовый без проверки результата.

## Связь с другими элементами продукта

Этот навык должен применяться вместе:

- с принципом корпуса первичных данных;
- с контрольным слоем, если создаётся или меняется переносимый элемент продукта;
- с навыком аналитического разбора, если нужно понять, достаточно ли оснований
  для нового навыка;
- с навыком технического письма, если утверждённое содержание нужно оформить в
  понятный документ;
- с ролью аналитика, если нужно отделить повторяемую проблему от локального
  случая.
