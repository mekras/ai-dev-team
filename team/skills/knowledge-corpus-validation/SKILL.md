# Навык: проверка корпуса знаний

## Назначение

Этот навык задаёт порядок комплексной проверки корпуса знаний проекта на
формальные ошибки, полноту обработки первичных данных, соответствие источников и
извлечённых утверждений специфике проекта, а также на отсутствие внутренних
противоречий.

Навык нужен для того, чтобы проект мог поддерживать качество и надёжность
корпуса знаний как основания для принятия решений, проектирования и
документирования.

## Когда применять навык

Навык применяется:

- после внесения изменений в корпус знаний;
- если пользователь попросил проверить корпус знаний.

Навык не применяется для единичной проверки одного файла или одного источника,
если это не часть общей проверки корпуса.

## Входные данные

Для применения навыка должны быть доступны:

- структура каталогов корпуса знаний;
- перечень источников и их статусы;
- нормализованные представления первичных данных;
- извлечённые утверждения;
- производные документы проекта;
- сведения о специфике проекта (цель, область применения, ограничения).

Если этих данных недостаточно, ограничение должно быть зафиксировано явно.

## Результат применения навыка

Результатом должен быть отчёт пользователю о проверке корпуса знаний, по
которому понятно:

- какие формальные ошибки найдены и где именно;
- какие источники не доведены до конца цепочки обработки;
- какие источники или утверждения не соответствуют специфике проекта;
- какие внутренние противоречия выявлены;
- насколько корпус надёжен как основание для проектной работы.

В отчёте не надо выводить лишние сведения, такие как списки проверенных объектов
или объекты без проблем.

В конце отчёта предложи начать устранение проблем по одной, начиная с первой.

## Порядок применения

### 1. Проверить формальную целостность корпуса

Нужно проверить структуру и состав корпуса на соответствие ожидаемой схеме.

Проверь:

- наличие всех обязательных файлов в каждом источнике:
  - `source.md` — описание источника;
  - `status.json` — статус обработки (если применимо);
  - `page-index.tsv` — индекс страниц (для многофайловых источников);
- наличие файла `.gitignore` в источниках, которые не должны отслеживаться;
- правильность учёта `.gitignore` для файлов из `source.md`:
  - если `source.md` ссылается на игнорируемые файлы Git, их отсутствие в
    рабочей копии после клонирования не считать формальной ошибкой;
  - такие артефакты считать восстановимыми и проверять по статусу нормализации;
- соответствие структуры каталогов ожидаемой иерархии:
  - `data/primary/` — первичные данные;
  - `data/normalized/` — нормализованные представления;
  - `data/statements/` — извлечённые утверждения;
  - `data/inventory/` — реестр источников;
- отсутствие лишних файлов, не относящихся к корпусу;
- корректность имён файлов и каталогов (отсутствие пробелов, специальных
  символов, нечитаемых имён);
- корректность формата файлов (Markdown, JSON, TSV) через синтаксическую
  проверку.

Зафиксируй все найденные формальные ошибки в отчёте.

### 2. Проверить полноту цепочки обработки первичных данных

Для каждого источника проверь проход по всей цепочке обработки.

Проверь:

- для каждого источника в `data/primary/`:
  - существует ли описание в `data/inventory/`;
  - существует ли нормализованное представление в `data/normalized/`;
  - существует ли файл извлечённых утверждений в `data/statements/`;
- для многофайловых источников:
  - все ли страницы из `page-index.tsv` присутствуют в `data/primary/`;
  - все ли страницы нормализованы;
  - все ли страницы отражены в утверждениях;
  - если страницы нормализованы в игнорируемой папке (`pages/` и т. п.),
    зафиксировать их как «нормализованный слой нужно восстановить после
    клонирования».
- статусы обработки в `status.json`:
  - нет ли зависших статусов;
  - нет ли расхождений между статусом и фактическим наличием файлов;
- свежие ли данные:
  - когда источник последний раз синхронизировался;
  - не устарел ли нормализованный слой;
  - не устарели ли извлечённые утверждения.

Зафиксируй источники, которые не прошли полную цепочку обработки.

### 3. Проверить соответствие источников специфике проекта

Для каждого источника проверь, относится ли он к проекту.

Проверь:

- связь источника с целью проекта:
  - описывает ли источник предметную область проекта;
  - описывает ли источник методологию или практику, которую проект использует;
  - является ли источник внешним основанием для навыка или правила;
- тип источника:
  - является ли источник первичным материалом, а не пересказом;
  - является ли источник авторитетным и надёжным;
  - не является ли источник устаревшим для целей проекта;
- условия использования:
  - зафиксирована ли лицензия или условия использования;
  - разрешает ли лицензия нужное использование в проекте;
  - соблюдены ли требования лицензии в корпусе;
- видимость проекта:
  - соответствует ли способ хранения источника видимости проекта;
  - не попадает ли ограниченный материал в публичный Git.

Зафиксируй источники, которые не соответствуют специфике проекта или требуют
перепроверки.

### 4. Проверить соответствие извлечённых утверждений специфике проекта

Для каждого файла извлечённых утверждений проверь содержание.

Проверь:

- связь с источником:
  - можно ли проследить каждое утверждение до первичного материала;
  - не потеряна ли связь с источником;
  - нет ли утверждений без основания;
- статус утверждений:
  - разделены ли факты, наблюдения и выводы;
  - не выдан ли вывод за факт;
  - не превращено ли единичное наблюдение в общее правило;
- соответствие специфике проекта:
  - относится ли утверждение к области проекта;
  - не извлечено ли утверждение, не имеющее отношения к проекту;
  - не упущены ли важные для проекта утверждения;
- качество извлечения:
  - нет ли расплывчатых формулировок, которые нельзя проверить;
  - не сглажена ли неоднозначность источника;
  - зафиксированы ли ограничения и сомнения.

Зафиксируй утверждения, которые не соответствуют специфике проекта или извлечены
с ошибками.

### 5. Проверить корпус на внутренние противоречия

Сопоставь утверждения из разных источников между собой.

Проверь:

- явные противоречия:
  - утверждают ли источники противоположное об одном и том же;
  - есть ли конфликт между правилом и фактом;
  - есть ли конфликт между решением и ограничением;
- скрытые противоречия:
  - ведут ли разные утверждения к разным выводам в одной ситуации;
  - есть ли терминологический конфликт (одно слово, разные значения);
  - есть ли конфликт между переносимым ядром и локальными особенностями;
- противоречия между слоями:
  - противоречит ли извлечённое утверждение первичному источнику;
  - противоречит ли производный документ извлечённым утверждениям;
  - противоречит ли нормализованный слой первичному материалу;
- противоречия во времени:
  - не устарело ли правило из-за нового источника;
  - не противоречит ли свежий источник старому решению.

Для каждого противоречия зафиксируй:

- какие утверждения или источники противоречат;
- в чём именно заключается противоречие;
- насколько противоречие существенно для проекта;
- требуется ли решение человека для разрешения.

## Что считается ошибкой применения навыка

- формальная ошибка пропущена из-за невнимательности;
- источник считается прошедшим обработку без проверки всех файлов;
- утверждение принято как соответствующее проекту без проверки связи с
  источником;
- противоречие сглажено вместо явной фиксации;
- проверка остановлена на одном слое без перехода к другим;
- отчёт не даёт явного перечня проблем и рекомендаций;
- слабость корпуса скрыта вместо явной фиксации.

## Связь с другими элементами продукта

Этот навык должен применяться вместе:

- с принципом корпуса первичных данных;
- с ролью `source-inventory`;
- с ролью аналитика для разбора противоречий;
- с навыком приёма первичных данных в корпус;
- с навыком нормализации первичных данных;
- с навыком извлечения утверждений из данных;
- с контрольным слоем, если проверка выявляет существенные проблемы корпуса;
- с навыком аналитического разбора для оценки влияния противоречий.
