# Навык: настройка проекта

## Назначение

Навык помогает менеджеру проекта настроить новый проект или перенастроить
существующий проект под актуальные требования и правила `ai-dev-team` через
короткий диалог и минимальные безопасные изменения.

Навык нужен, чтобы агент не настраивал проект случайным шаблоном, не выдумывал
лицензию, не перезаписывал существующие файлы, не оставлял устаревшие правила
подключения продукта и сразу предлагал рациональное использование подагентов.

## Когда применять

Применяй навык, если пользователь просит:

- настроить проект;
- перенастроить существующий проект;
- обновить проект под актуальные правила `ai-dev-team`;
- начать новый проект;
- инициализировать проект;
- подготовить стартовые файлы проекта;
- завести базовый `README.md`, `LICENSE`, `AGENTS.md` и `.gitignore`;
- подготовить новый репозиторий к работе с агентами.

Не применяй навык для обычной правки одного существующего файла, настройки
только подагентов, создания одной лицензии или полного проектирования
документации.

## Режим по умолчанию

По умолчанию работай в диалоговом режиме.

Сначала проверь текущий каталог и существующие файлы. Если файлы уже есть,
прочитай их перед изменением и не перезаписывай без явного решения человека. Для
существующего проекта сначала классифицируй состояние: чего не хватает, какие
правила устарели и какие изменения можно сделать без смены смысла проекта.

Если сведений не хватает, веди пошаговый диалог: задавай ровно один вопрос за
ход и жди ответа человека. Не объединяй несколько обязательных вопросов в одно
сообщение, кроме случая, когда человек прямо просит дать анкету или сразу
предоставил сведения для нескольких решений. Не создавай окончательные файлы,
пока не известны обязательные решения:

- название и краткое назначение проекта;
- публичный или закрытый проект;
- владелец или автор для `LICENSE`;
- выбранная лицензия или решение «пока не выбирать лицензию»;
- используется ли Git;
- должен ли проект использовать `ai-dev-team` через APM.

## Обязательный порядок

1. Определи, есть ли уже Git и стартовые файлы: `README.md`, `LICENSE`,
   `AGENTS.md`, `.gitignore`.
2. Для существующего проекта прочитай найденные файлы и сравни их с актуальными
   правилами продукта. Зафиксируй, какие изменения являются обновлением
   настройки, а какие требуют отдельного решения человека.
3. Собери недостающие сведения через `references/setup-dialogue.md`: один вопрос
   за ход, начиная с самого блокирующего решения.
4. Зафиксируй минимальный состав создаваемых или обновляемых файлов.
5. Создай или обнови `README.md` через навык `project-readme`.
6. Создай `LICENSE` только на основании выбранной лицензии или явного решения
   человека. Не подставляй MIT, Apache, GPL или проприетарный текст по
   умолчанию.
7. Создай или обнови `AGENTS.md` как минимальные правила проекта для агента:
   подключение `ai-dev-team`, видимость проекта, расположение корпуса знаний,
   базовые ограничения и порядок проверки.
8. Если проект использует Git, создай или обнови `.gitignore` минимально под
   известный стек и общие локальные артефакты. Если стек неизвестен, используй
   только универсальные безопасные исключения.
9. Предложи настроить рациональное использование подагентов. Если пользователь
   подтверждает внедрение, передай задачу в `subagent-model-routing` в режиме
   `implementation`; если подтверждения нет, зафиксируй предложение как
   следующий шаг.
10. Предложи дополнительные стартовые файлы только как отдельные варианты:
    `.editorconfig`, `docs/concept.md`, `requirements.md`, начальный
    `knowledge/` или приватный локальный слой знаний.
11. Перед завершением проверь, какие файлы созданы или изменены, и добавь новые
    отслеживаемые файлы в индекс Git, если проект использует Git и правила
    проекта не запрещают индексацию.

## Шаблоны

Используй `assets/minimal-files.md` как рабочую заготовку. Адаптируй её под
собранные сведения и не оставляй в итоговых файлах плейсхолдеры без явного
статуса открытого вопроса.

## Границы

- Не создавай крупную структуру каталогов без отдельного обоснования.
- Не выдумывай концепцию, требования, архитектуру, стек, сроки, роли и
  бизнес-эффект.
- Не записывай настоящие секреты в `AGENTS.md`, `.env`, локальные правила или
  другие текстовые файлы.
- Не создавай `.gitignore`, если проект не использует Git и человек не просит
  подготовить его заранее.
- Не добавляй ссылку на приватный локальный файл в отслеживаемый `AGENTS.md`,
  если человек требует скрыть сам факт локального слоя.

## Результат

Результатом работы должны быть:

- минимальные стартовые файлы проекта или вопросы, без которых их нельзя создать
  безопасно;
- явный список созданных и изменённых файлов;
- список решений человека, которые были использованы;
- предложение по рациональному использованию подагентов;
- открытые вопросы и следующие безопасные шаги.
