# Навык: сопровождение проектного README

## Назначение

Навык задаёт порядок создания, изменения и проверки главного `README.md`
проекта.

Главный `README.md` является пользовательской точкой входа в проект. Он не
должен подменяться спецификацией, архитектурным описанием, журналом решений или
документацией реализации.

## Когда применять навык

Навык должен применяться, если нужно:

- создать новый `README.md`;
- изменить существующий `README.md`;
- проверить качество `README.md`.

Навык не должен применяться для второстепенных файлов `README.md` проекта, а
также `AGENTS.md`, `CHANGELOG.md` и других файлов.

## Входные данные

Для применения навыка должны быть доступны:

- понимание смысла и характера проекта;
- концепция проекта или явное понимание, что концепция ещё не создана;
- понимание, кто основной пользователь `README.md` и какие у него интересы;
- сведения о первом практическом действии, которое пользователь должен смочь
  выполнить;
- проверяемые факты о подключении, запуске, использовании или разработке
  проекта.

Если этих сведений недостаточно, их нужно сначала найти в проекте или запросить
у пользователя.

Если в проекте есть концепция, `README.md` не должен расходиться с ней. Если
концепции нет, зафиксируй это как ограничение и не выдумывай её внутри README.

## Порядок применения

### 1. Определить пользователя README

Нужно явно понять:

- для кого пишется `README.md`;
- за чем этот пользователь пришёл;
- какое первое практическое действие он должен смочь выполнить после чтения.

Перед изменением `README.md` агент должен отдельно ответить:

- кто основной читатель документа;
- какое первое действие он должен выполнить;
- какие разделы помогают этому действию;
- какие разделы являются внутренними подробностями и должны быть вынесены.

Структура `README.md` не должна выбираться по шаблону заранее. Агент должен
сначала определить тип проекта и основное первое действие пользователя, а затем
выбрать минимальную структуру под этот сценарий.

Типовые первые действия:

- подготовить систему к использованию продукта;
- установить и запустить приложение;
- использовать библиотеку в коде;
- развернуть сервис;
- открыть документацию или спецификацию;
- начать разработку самого проекта.

### 2. Собрать только нужные факты

Перед правкой нужно собрать подтверждённые сведения:

- что это за проект;
- в чём его смысл, для чего он нужен;
- что он даёт пользователю;
- как начать его использование;
- что потребуется пользователю в дальнейшем.

Все факты должны опираться только на подтверждённые сведения.

Если есть концепция проекта, смысл и границы проекта для `README.md` должны
опираться на неё. Если концепция противоречит другим источникам, не сглаживай
конфликт: зафиксируй его и вынеси решение человеку.

Если сведения нужны разработчику проекта, но не нужны пользователю для первого
практического действия, их нужно вынести в отдельный документ и дать ссылку
после основного сценария.

Для каждой команды, которую агент собирается включить в `README.md`, нужно
отдельно определить и зафиксировать область действия команды. Минимальные
классы:

- установка пакета или продукта;
- настройка клиента или среды пользователя;
- подключение конкретного проекта;
- проверка установки или подключения;
- локальная разработка или сопровождение.

Классификация должна опираться не на название команды, а на проверяемый эффект:
какие файлы, каталоги, настройки или артефакты она изменяет и в какой области.
Если команда пишет в домашний каталог пользователя, глобальную конфигурацию,
настройки клиента или другой ресурс вне текущего проекта, её нельзя описывать
как шаг подключения конкретного проекта.

### 3. Выбрать структуру под сценарий

`README.md` должен закрыть пользовательские вопросы:

- что это за проект (кратко), в чём его смысл;
- что он даёт своим пользователям;
- как установить, подключить или начать применение;
- какое первое действие выполнить после установки или подключения;
- где найти подробную документацию.

Раздел разработки добавляется только если `README.md` адресован сопровождающим
проекта или пользователь явно просит описать разработку.

Нельзя делать структуру репозитория основной формой `README.md`. Разделы про
каталоги, внутренние инструменты и служебные файлы не должны идти раньше
назначения проекта, пользы и способа применения.

Если раздел называется «Как пользоваться», он должен описывать пользовательский
сценарий, а не внутренние компоненты.

### 4. Разделить пользовательский и разработческий сценарии

Пользовательский сценарий должен содержать:

- подходящий для читателя способ установки, подключения или первого запуска;
- команду обновления, если она нужна пользователю;
- объяснение, что произойдёт после установки, подключения или запуска;
- первый рабочий шаг после установки, подключения или запуска.

Разработческий сценарий должен содержать локальные проверки, структуру проекта и
команды сопровождения только в отдельном разделе разработки.

Команды установки и первого запуска должны соответствовать сценарию читателя.
Если документ адресован внешнему пользователю, основной путь должен быть
воспроизводим вне локального окружения автора. Локальные пути допустимы, когда
`README.md` описывает локальную разработку, внутреннюю установку или явно
ограниченный сценарий.

### 5. Внести правки в файл

- Внести в `README.md` требуемые изменения, используя навыки `ru-dev`, `ru-lang`
  и `ru-text` при их наличии.

### 6. Проверить README как пользовательский вход

После правки нужно перечитать `README.md` как новый пользователь и проверить:

- понятно ли за первую минуту, что это за проект;
- понятно ли, что она даёт пользователям;
- не расходится ли описание проекта с концепцией, если она есть;
- есть ли рабочая команда установки, подключения или первого запуска;
- не попали ли внутренние проверки в пользовательский сценарий;
- не описывает ли документ структуру раньше пользы;
- все ли команды имеют понятную аудиторию.
- для каждой ли команды явно понятно, что именно она меняет: систему
  пользователя, клиент агента, текущий проект или только проверку состояния.

## Результат применения навыка

Результатом должен быть `README.md`, который:

- опирается на проверяемые факты;
- ориентирован на пользователя проекта;
- ведёт к первому практическому действию;
- даёт понятный маршрут следующего шага.

## Что считается ошибкой применения навыка

- В документ попали непроверенные команды, статусы или сценарии использования.
- Документ содержит требования вместо руководства для пользователя.
- Документ объясняет внутренний протокол раньше пользовательского результата.
- Документ смешивает основной сценарий с диагностикой и разработкой.
- Документ заставляет читать несколько документов до первого полезного действия.
- Документ описывает структуру проекта раньше назначения и пользы.
- Пользовательский быстрый старт содержит команды сопровождения проекта.
- Команды установки или первого запуска не соответствуют аудитории документа.
- Команда отнесена к установке, подключению проекта или проверке только по
  названию, а не по фактической области действия.
- Заголовок раздела обещает пользу или использование, но внутри описан состав
  файлов или внутренние проверки.

## Связь с другими элементами продукта

Этот навык должен применяться вместе:

- с ролью технического писателя;
- с навыком технического письма как общим основанием;
- с документами о подключении, запуске или первом применении, если `README.md`
  описывает использование продукта в проекте;
- с контрольным слоем, если изменение `README.md` влияет на значимый
  пользовательский источник истины.

## Внешние основания навыка

Для этого навыка уместно использовать публичные авторитетные материалы по
проектной и пользовательской документации.

В частности:

- руководства сообщества `Write the Docs`.

Публичный источник помогает обосновать правило навыка, но не заменяет проверку
того, что именно нужно пользователю конкретного проекта.
