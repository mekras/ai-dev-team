# Проверка: различие установки и подключения проекта

## Ситуация

Агенту поручили переписать главный `README.md` продукта, который подключается к
проектам через локальную APM-копию и переносимые машинные инструкции,
скомпилированные APM в файлы конкретных клиентов.

В проекте актуальны команды:

- `apm install github.com/mekras/ai-dev-team#master --target claude`;
- `apm install github.com/mekras/ai-dev-team#master --target codex`;
- `apm compile --single-agents --target claude`;
- `apm compile --single-agents --target codex`.

Проект больше не поддерживает глобальную установку, собственный CLI, каналы
подключения и отдельный серверный транспорт. Подключение целевого проекта
выполняется штатными средствами APM.

## Ожидаемое поведение

Агент должен:

- определить основного читателя как пользователя продукта;
- проверить по коду или достоверной документации, что делает каждая команда;
- проверить, какой способ установки является актуальным после перехода на `APM`;
- отказаться от глобальной установки продукта как основного пользовательского
  сценария;
- не возвращать в README команды удалённого CLI, отдельного серверного
  транспорта или глобальной установки;
- отнести `apm install github.com/mekras/ai-dev-team#master --target claude` и
  `apm install github.com/mekras/ai-dev-team#master --target codex` к
  подключению продукта в целевом проекте штатными средствами APM;
- отнести `apm compile --single-agents --target claude` и
  `apm compile --single-agents --target codex` к созданию клиентских инструкций
  для явно выбранной цели;
- требовать явный выбор одной цели: `claude` или `codex`;
- объяснить, что продукт устанавливается в каждый целевой проект локально через
  APM;
- использовать наличие APM-зависимости и успешную компиляцию инструкций как
  проверку подключения;
- построить разделы так, чтобы установка клиента и подключение проекта не
  смешивались.

## Ошибочное поведение

Агент не должен:

- относить команду к разделу только по словам `setup`, `connect`, `install` или
  `agents` в названии;
- оставлять глобальную npm-установку основным способом установки после
  APM-миграции без проверки, что это всё ещё актуальный публичный путь;
- оставлять глобальную npm-установку как переходный путь после решения о
  локальной установке продукта в целевой проект через APM;
- оставлять глобальную APM-установку основным способом установки после решения
  подключать продукт штатными средствами APM в целевом проекте;
- выдавать `npm install` или `apm install --target ...` из корня репозитория
  за установку продукта для внешнего пользователя;
- использовать собственный CLI продукта как основной путь, если подключение
  должно выполняться средствами APM;
- подключать продукт сразу для `claude,codex` или `all`, если пользователь
  должен явно выбрать один клиентский инструмент;
- опускать `--target` в пользовательских командах подключения и компиляции;
- помещать команды удалённого CLI или отдельного серверного транспорта в раздел
  подключения конкретного проекта;
- требовать ручной правки `CLAUDE.md` как единственного источника проектных
  правил, если правило должно работать и в Codex, и в Claude;
- описывать изменение `~/.codex/config.toml`, `~/.claude.json` или
  `~/.claude/CLAUDE.md` как изменение текущего проекта;
- смешивать настройку клиента, проверку и подключение проекта без явного
  разделения по области действия.

## Критерий прохождения

Новый пользователь за первую минуту чтения понимает:

- какие команды подключают продукт именно к текущему проекту;
- где явно выбирается клиентский инструмент: Claude или Codex;
- как проверить, что установка и подключение выполнены правильно.
