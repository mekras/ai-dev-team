# Проверка: README как пользовательская точка входа

## Ситуация

Агенту поручили написать или изменить главный `README.md` проекта. В проекте
есть внутренние документы с требованиями, архитектурными решениями,
интеграционными протоколами и командами разработки.

## Ожидаемое поведение

Агент должен:

- определить основного читателя `README.md`;
- определить первое практическое действие читателя;
- поместить это действие в основной пользовательский сценарий;
- вынести внутренние требования, протоколы и подробности разработки в отдельные
  документы или разделы после основного сценария;
- добавить проверку результата первого действия, если она известна;
- сохранить ссылки на подробные документы, не заставляя читать их до первого
  полезного действия.

## Ошибочное поведение

Агент не должен:

- выбирать структуру `README.md` по заранее заданному шаблону без учёта проекта;
- описывать требования к интеграции вместо шагов пользователя;
- предлагать ручной запуск фонового сервера как проверку, если команда выглядит
  зависшей;
- объяснять внутренний протокол раньше результата для пользователя;
- смешивать подключение, диагностику, разработку и архитектуру в один основной
  сценарий.

## Критерий прохождения

Новый пользователь за первую минуту чтения понимает:

- что это за проект;
- зачем он ему нужен;
- какое первое действие выполнить;
- как понять, что первое действие сработало;
- куда перейти за подробностями после основного сценария.
