# Извлечённые утверждения: `RADR`

## Основание

- Источник учёта: `knowledge/inventory/RADR.md`
- Первичный источник: `knowledge/primary/RADR/source.md`

- Дата извлечения: `2026-06-24`

## Назначение

Этот документ фиксирует набор утверждений по книге `RADR`, достаточный для того,
чтобы роли команды (прежде всего архитектор и разработчик, а также технический
писатель и специалист по безопасности) уверенно и правильно работали с
проектированием веб-ориентированных REST API, а также для проверки влияния на
навыки и документы проекта.

Утверждения не являются требованиями, правилами или решениями для этого
репозитория. Они фиксируют наблюдаемое содержание книги. Книга формулирует
правила с модальностью must / should / may; в утверждениях она передана словами
«обязательно», «следует» и «допустимо».

Каждое утверждение опирается на локальную текстовую выгрузку
`knowledge/primary/RADR/files/rest-api-design-rulebook.txt`, полученную из PDF
утилитой `pdftotext -layout`. Полные артефакты являются локальными
неотслеживаемыми данными. Все утверждения имеют статус `ready_for_review`.
Английские технические термины и коды состояний сохранены как есть. Дословные
крупные фрагменты не приводятся: содержание пересказано.

## Глава 1. Introduction

### RADR-001. Шесть ограничений архитектуры Веба

- Утверждение: архитектурный стиль Веба образован шестью группами ограничений:
  client-server, uniform interface, layered system, cache, stateless и
  code-on-demand.
- Фрагмент источника: глава 1 «Introduction», раздел «Web Architecture».
- Область применения: архитектор.

### RADR-002. Code-on-demand — единственное опциональное ограничение

- Утверждение: code-on-demand — единственное необязательное ограничение стиля
  Веба, остальные пять обязательны для соответствия.
- Фрагмент источника: глава 1 «Introduction», раздел «Code-On-Demand».
- Область применения: архитектор.

### RADR-003. Четыре ограничения единообразного интерфейса

- Утверждение: единообразный интерфейс задаётся четырьмя ограничениями:
  идентификация ресурсов, манипуляция через представления, самоописываемые
  сообщения и HATEOAS.
- Фрагмент источника: глава 1 «Introduction», раздел «Uniform Interface».
- Область применения: архитектор.

### RADR-004. Stateless как фактор масштабируемости

- Утверждение: сервер не обязан запоминать состояние клиентов, поэтому клиент
  должен передавать весь нужный контекст в каждом запросе, что повышает
  масштабируемость.
- Фрагмент источника: глава 1 «Introduction», раздел «Stateless».
- Область применения: архитектор.

### RADR-005. Модель зрелости Ричардсона

- Утверждение: модель зрелости Ричардсона из трёх уровней (URI, HTTP,
  Hypermedia) служит способом оценки «RESTful» дизайна API.
- Фрагмент источника: глава 1 «Introduction» / предисловие, упоминание Леонарда
  Ричардсона.
- Область применения: архитектор.

### RADR-006. WRML как рамка моделирования

- Утверждение: автор вводит WRML (Web Resource Modeling Language) и медиатип
  `application/wrml` с подключаемыми компонентами format и schema для
  единообразного описания форм представлений.
- Фрагмент источника: глава 1 «Introduction», раздел «WRML».
- Область применения: архитектор.

## Глава 2. Identifier Design with URIs

### RADR-007. Прямой слеш как иерархический разделитель

- Утверждение: прямой слеш `/` обязательно используется для обозначения
  иерархических отношений между ресурсами в пути URI.
- Фрагмент источника: глава 2, правило «Forward slash separator».
- Область применения: разработчик.

### RADR-008. Без завершающего слеша

- Утверждение: завершающий прямой слеш `/` в URI использоваться не должен.
- Фрагмент источника: глава 2, правило «A trailing forward slash should not be
  included».
- Область применения: разработчик.

### RADR-009. Дефисы для читаемости

- Утверждение: для улучшения читаемости URI следует использовать дефисы `-`.
- Фрагмент источника: глава 2, правило «Hyphens should be used».
- Область применения: разработчик.

### RADR-010. Без подчёркиваний

- Утверждение: подчёркивания `_` в URI использоваться не должны, так как
  подчёркивание ссылок может их скрыть.
- Фрагмент источника: глава 2, правило «Underscores should not be used».
- Область применения: разработчик.

### RADR-011. Нижний регистр в путях

- Утверждение: в путях URI следует предпочитать строчные буквы.
- Фрагмент источника: глава 2, правило «Lowercase letters should be preferred».
- Область применения: разработчик.

### RADR-012. Без расширений файлов в URI

- Утверждение: расширения файлов в URI включаться не должны; для выбора формата
  следует использовать согласование медиатипа.
- Фрагмент источника: глава 2, правило «File extensions should not be included».
- Область применения: разработчик.

### RADR-013. Согласованные субдомены

- Утверждение: домен API следует дополнять субдоменом `api` (например,
  `api.soccer.restapi.org`), а портал разработчика — субдоменом `developer`.
- Фрагмент источника: глава 2, правила «Consistent subdomain names».
- Область применения: архитектор.

### RADR-014. Существительное в единственном числе для document

- Утверждение: ресурс-документ (document) следует именовать существительным или
  именной группой в единственном числе.
- Фрагмент источника: глава 2, правило «A singular noun should be used for
  document names».
- Область применения: разработчик.

### RADR-015. Множественное число для collection

- Утверждение: коллекцию (collection) следует именовать существительным во
  множественном числе, отражающим содержимое.
- Фрагмент источника: глава 2, правило «A plural noun should be used for
  collection names».
- Область применения: разработчик.

### RADR-016. Множественное число для store

- Утверждение: хранилище (store) следует именовать существительным во
  множественном числе.
- Фрагмент источника: глава 2, правило «A plural noun should be used for store
  names».
- Область применения: разработчик.

### RADR-017. Глагол для controller

- Утверждение: ресурс-контроллер (controller) следует именовать глаголом или
  глагольной группой, обозначающей действие (например, `register`, `dedupe`).
- Фрагмент источника: глава 2, правило «A verb or verb phrase should be used for
  controller names».
- Область применения: разработчик.

### RADR-018. Запрет имён CRUD-функций в URI

- Утверждение: имена CRUD-функций в URI использоваться не должны; функцию должен
  обозначать HTTP-метод, а не путь (анти-паттерн `GET /deleteUser?id=1234`).
- Фрагмент источника: глава 2, правило «CRUD function names should not be used
  in URIs».
- Область применения: разработчик.

### RADR-019. URI — единственные значимые идентификаторы

- Утверждение: клиенты должны считать URI единственными осмысленными
  идентификаторами ресурсов; внутренние ID бэкенда для клиента бессмысленны.
- Фрагмент источника: глава 2, правило «Variable path segments may be
  substituted with identity-based values».
- Область применения: архитектор.

### RADR-020. Query для фильтрации

- Утверждение: компонент query допустимо использовать для фильтрации коллекций и
  хранилищ (например, `GET /users?role=admin`).
- Фрагмент источника: глава 2, правило «The query component may be used to
  filter collections».
- Область применения: разработчик.

### RADR-021. Query для пагинации

- Утверждение: для пагинации результатов следует использовать query-параметры
  `pageSize` (максимум элементов) и `pageStartIndex` (нулевой индекс первого
  элемента).
- Фрагмент источника: глава 2, правило «The query component should be used to
  paginate collections».
- Область применения: разработчик.

### RADR-022. Сложную фильтрацию — через controller с телом запроса

- Утверждение: когда сложность фильтрации или пагинации превышает возможности
  query, следует завести специальный controller (например,
  `POST /users/search`), принимающий параметры в теле запроса.
- Фрагмент источника: глава 2, раздел «URI Query Design».
- Область применения: архитектор.

## Глава 3. Interaction Design with HTTP

### RADR-023. Запрет туннелирования методов

- Утверждение: GET и POST не должны использоваться для туннелирования других
  методов запроса, чтобы не нарушать прозрачность протокола.
- Фрагмент источника: глава 3, правило «GET and POST must not be used to
  tunnel».
- Область применения: разработчик.

### RADR-024. GET для получения представления

- Утверждение: GET обязательно используется для получения представления
  состояния ресурса; запрос GET не должен иметь тела и должен быть безопасным
  (без побочных эффектов).
- Фрагмент источника: глава 3, правило «GET must be used to retrieve a
  representation».
- Область применения: разработчик.

### RADR-025. HEAD для заголовков

- Утверждение: HEAD следует использовать для получения тех же заголовков, что и
  GET, но с пустым телом — например, чтобы проверить существование ресурса или
  прочитать метаданные.
- Фрагмент источника: глава 3, правило «HEAD should be used to retrieve response
  headers».
- Область применения: разработчик.

### RADR-026. PUT для вставки и обновления в store

- Утверждение: PUT обязательно используется и для добавления нового ресурса в
  store по заданному клиентом URI, и для обновления или замены уже хранимого
  ресурса.
- Фрагмент источника: глава 3, правило «PUT must be used to both insert and
  update a stored resource».
- Область применения: разработчик.

### RADR-027. PUT для изменения mutable-ресурсов

- Утверждение: для внесения изменений в изменяемые ресурсы клиенты обязательно
  используют метод PUT с телом, отражающим желаемые изменения.
- Фрагмент источника: глава 3, правило «PUT must be used to update mutable
  resources».
- Область применения: разработчик.

### RADR-028. POST для создания в collection

- Утверждение: POST обязательно используется для создания нового ресурса внутри
  коллекции; тело содержит предлагаемое начальное состояние ресурса.
- Фрагмент источника: глава 3, правило «POST must be used to create a new
  resource in a collection».
- Область применения: разработчик.

### RADR-029. POST для запуска контроллеров

- Утверждение: POST обязательно используется для вызова контроллеров; метод
  неидемпотентен и небезопасен, поэтому подходит для операций, не отображаемых
  на другие методы.
- Фрагмент источника: глава 3, правило «POST must be used to execute
  controllers».
- Область применения: разработчик.

### RADR-030. DELETE для удаления

- Утверждение: DELETE обязательно используется для полного удаления ресурса из
  родителя; после удаления любой GET/HEAD к нему должен возвращать 404. «Мягкое»
  удаление следует делать через controller и POST.
- Фрагмент источника: глава 3, правило «DELETE must be used to remove a
  resource».
- Область применения: разработчик.

### RADR-031. OPTIONS для метаданных взаимодействий

- Утверждение: OPTIONS следует использовать для получения метаданных о доступных
  взаимодействиях с ресурсом, включая заголовок `Allow` (например,
  `Allow: GET, PUT, DELETE`).
- Фрагмент источника: глава 3, правило «OPTIONS should be used to retrieve
  metadata».
- Область применения: разработчик.

### RADR-032. 200 для неспецифического успеха

- Утверждение: код 200 («OK») следует использовать для обозначения успеха, когда
  нет более точного кода 2xx; ответ 200 должен включать тело.
- Фрагмент источника: глава 3, правило «200 should be used to indicate
  nonspecific success».
- Область применения: разработчик.

### RADR-033. 200 не для ошибок в теле

- Утверждение: код 200 не должен использоваться для передачи ошибок в теле
  ответа.
- Фрагмент источника: глава 3, правило «200 must not be used to communicate
  errors».
- Область применения: разработчик.

### RADR-034. 201 для успешного создания ресурса

- Утверждение: код 201 («Created») обязательно используется при создании ресурса
  в collection, добавлении в store или создании ресурса в результате действия
  контроллера.
- Фрагмент источника: глава 3, правило «201 must be used to indicate successful
  resource creation».
- Область применения: разработчик.

### RADR-035. 202 для асинхронного старта

- Утверждение: код 202 («Accepted») обязательно используется для обозначения
  успешного запуска асинхронной операции; его могут отправлять контроллеры, но
  не другие типы ресурсов.
- Фрагмент источника: глава 3, правило «202 must be used to indicate successful
  start of an asynchronous action».
- Область применения: разработчик.

### RADR-036. 204 для пустого тела

- Утверждение: код 204 («No Content») следует использовать, когда тело ответа
  намеренно пустое — обычно при PUT, POST или DELETE.
- Фрагмент источника: глава 3, правило «204 should be used when the response
  body is intentionally empty».
- Область применения: разработчик.

### RADR-037. 301 для постоянного перемещения

- Утверждение: код 301 («Moved Permanently») следует использовать при
  значительной переработке модели ресурсов с присвоением нового постоянного URI,
  указанного в заголовке `Location`.
- Фрагмент источника: глава 3, правило «301 should be used to relocate
  resources».
- Область применения: архитектор.

### RADR-038. 302 не использовать

- Утверждение: код 302 («Found») использоваться не должен из-за исторически
  неоднозначной семантики; вместо него следует применять 303 или 307.
- Фрагмент источника: глава 3, правило «302 should not be used».
- Область применения: разработчик.

### RADR-039. 303 для отсылки к другому URI

- Утверждение: код 303 («See Other») следует использовать, чтобы направить
  клиента к другому URI (например, после завершения работы контроллера) вместо
  нежелательного тела ответа.
- Фрагмент источника: глава 3, правило «303 should be used to refer the client
  to a different URI».
- Область применения: разработчик.

### RADR-040. 304 для экономии трафика

- Утверждение: код 304 («Not Modified») следует использовать для экономии полосы
  пропускания, сообщая клиенту, что его кэшированная копия актуальна.
- Фрагмент источника: глава 3, правило «304 should be used to preserve
  bandwidth».
- Область применения: разработчик.

### RADR-041. 307 для временного перенаправления

- Утверждение: код 307 («Temporary Redirect») следует использовать, чтобы
  сообщить клиенту о необходимости повторить запрос по другому URI с сохранением
  метода.
- Фрагмент источника: глава 3, правило «307 should be used to tell clients to
  resubmit».
- Область применения: разработчик.

### RADR-042. 400 для неспецифической ошибки клиента

- Утверждение: код 400 («Bad Request») допустимо использовать для обозначения
  неспецифической ошибки на стороне клиента, когда нет более точного кода 4xx.
- Фрагмент источника: глава 3, правило «400 may be used to indicate nonspecific
  failure».
- Область применения: разработчик.

### RADR-043. 401 для проблем аутентификации

- Утверждение: код 401 («Unauthorized») обязательно используется при проблеме с
  учётными данными клиента.
- Фрагмент источника: глава 3, правило «401 must be used when there is a problem
  with the client's credentials».
- Область применения: специалист по безопасности.

### RADR-044. 403 для запрета доступа

- Утверждение: код 403 («Forbidden») следует использовать для запрета доступа
  независимо от статуса аутентификации.
- Фрагмент источника: глава 3, правило «403 should be used to forbid access».
- Область применения: специалист по безопасности.

### RADR-045. 404 для ненайденного URI

- Утверждение: код 404 («Not Found») обязательно используется, когда URI клиента
  не может быть сопоставлен ни одному ресурсу.
- Фрагмент источника: глава 3, правило «404 must be used when a client's URI
  cannot be mapped».
- Область применения: разработчик.

### RADR-046. 405 для недопустимого метода

- Утверждение: код 405 («Method Not Allowed») обязательно используется, когда
  HTTP-метод не поддерживается ресурсом.
- Фрагмент источника: глава 3, правило «405 must be used when the HTTP method is
  not supported».
- Область применения: разработчик.

### RADR-047. 406 и 415 при проблемах с медиатипом

- Утверждение: код 406 («Not Acceptable») обязательно используется, когда
  запрошенный медиатип не может быть предоставлен; код 415 («Unsupported Media
  Type») — когда медиатип тела запроса не поддерживается.
- Фрагмент источника: глава 3, правила «406 must be used…» и «415 must be
  used…».
- Область применения: разработчик.

### RADR-048. 409 для конфликта состояния

- Утверждение: код 409 («Conflict») следует использовать для обозначения
  нарушения состояния ресурса (например, неоднозначность PUT в store).
- Фрагмент источника: глава 3, правило «409 should be used to indicate a
  violation of resource state».
- Область применения: разработчик.

### RADR-049. 412 для условных запросов

- Утверждение: код 412 («Precondition Failed») следует использовать для
  поддержки условных запросов, когда предусловие (например, `If-Match`) не
  выполнено.
- Фрагмент источника: глава 3, правило «412 should be used to support
  conditional operations».
- Область применения: разработчик.

### RADR-050. 500 для сбоя сервера

- Утверждение: код 500 («Internal Server Error») следует использовать для
  обозначения неисправности API на стороне сервера.
- Фрагмент источника: глава 3, правило «500 should be used to indicate API
  malfunction».
- Область применения: разработчик.

## Глава 4. Metadata Design

### RADR-051. Content-Type обязателен

- Утверждение: заголовок `Content-Type` обязательно используется для указания
  медиатипа тела сообщения.
- Фрагмент источника: глава 4, правило «Content-Type must be used».
- Область применения: разработчик.

### RADR-052. Last-Modified и ETag в ответах

- Утверждение: заголовки `Last-Modified` (метка времени изменения состояния) и
  `ETag` (непрозрачная версия представления) следует включать в ответы, особенно
  на GET; ETag нельзя генерировать из машинно-зависимых источников.
- Фрагмент источника: глава 4, правила «Last-Modified should be used», «ETag
  should be used».
- Область применения: разработчик.

### RADR-053. Условные PUT для store

- Утверждение: store обязательно должен поддерживать условные PUT-запросы,
  полагаясь на заголовки `If-Unmodified-Since` и/или `If-Match`; при
  несовпадении entity tag API обязан вернуть 412, при успехе — 200 или 204.
- Фрагмент источника: глава 4, правило «Stores must support conditional PUT
  requests».
- Область применения: разработчик.

### RADR-054. Location для нового ресурса

- Утверждение: заголовок `Location` обязательно используется для указания URI
  вновь созданного ресурса; в ответе 202 он может указывать на статус
  асинхронной операции.
- Фрагмент источника: глава 4, правило «Location must be used».
- Область применения: разработчик.

### RADR-055. Заголовки кэширования для поощрения кэша

- Утверждение: для поощрения кэширования следует использовать `Cache-Control` с
  `max-age`, а для совместимости с HTTP 1.0 — `Expires` и `Date`.
- Фрагмент источника: глава 4, правило «Cache-Control, Expires, and Date
  response headers should be used to encourage caching».
- Область применения: разработчик.

### RADR-056. Подавление кэша и предпочтение малого max-age

- Утверждение: чтобы запретить кэш, допустимо добавить
  `Cache-Control: no-cache, no-store` (плюс `Pragma: no-cache`, `Expires: 0`);
  но кэширование следует поощрять и предпочитать малый `max-age` директиве
  `no-cache`.
- Фрагмент источника: глава 4, правила «…may be used to discourage caching»,
  «Caching should be encouraged».
- Область применения: разработчик.

### RADR-057. Кэш-заголовки с 200 и негативное кэширование

- Утверждение: заголовки истечения следует ставить на успешные ответы 200
  (GET/HEAD), а опционально — на ответы 3xx и 4xx (негативное кэширование) для
  снижения нагрузки.
- Фрагмент источника: глава 4, правила «Expiration caching headers should be
  used with 200», «…may optionally be used with 3xx and 4xx».
- Область применения: разработчик.

### RADR-058. Кастомные заголовки не меняют поведение методов

- Утверждение: кастомные HTTP-заголовки не должны менять поведение HTTP-методов;
  важная для интерпретации информация должна быть в теле или URI, а не в
  кастомном заголовке.
- Фрагмент источника: глава 4, правило «Custom HTTP headers must not be used to
  change the behavior».
- Область применения: разработчик.

### RADR-059. Прикладные медиатипы

- Утверждение: следует использовать прикладные (application-specific) медиатипы
  (например, `application/wrml` или vendor-specific с префиксом `vnd`), несущие
  специфичные метаданные, а не только общие типы.
- Фрагмент источника: глава 4, правило «Application-specific media types should
  be used».
- Область применения: архитектор.

### RADR-060. Согласование медиатипа через Accept

- Утверждение: при наличии нескольких представлений следует поддерживать
  согласование медиатипа через заголовок `Accept` (клиент выбирает format и
  версию schema).
- Фрагмент источника: глава 4, правило «Media type negotiation should be
  supported».
- Область применения: разработчик.

### RADR-061. Выбор медиатипа через query-параметр

- Утверждение: допустимо поддерживать выбор медиатипа через query-параметр
  `accept`; это предпочтительнее виртуальных расширений `.xml`, но является
  формой туннелирования и требует осторожности.
- Фрагмент источника: глава 4, правило «Media type selection using a query
  parameter may be supported».
- Область применения: разработчик.

### RADR-062. Версионирование схем через отдельные URI

- Утверждение: разные версии схемы следует оформлять как отдельные документы с
  разными URI (суффикс версии, например `Player-2`), при этом текущая версия
  доступна и по URI без номера.
- Фрагмент источника: глава 4, раздел «Media Type Schema Versioning».
- Область применения: архитектор.

## Глава 5. Representation Design

### RADR-063. JSON как формат представления

- Утверждение: при отсутствии стандартного формата для типа ресурса для
  структурирования данных следует использовать JSON.
- Фрагмент источника: глава 5, правило «JSON should be supported for resource
  representation».
- Область применения: разработчик.

### RADR-064. JSON должен быть well-formed

- Утверждение: JSON обязан быть корректным — имена всегда в двойных кавычках;
  имена следует писать в смешанном нижнем регистре (camelCase) и избегать
  спецсимволов.
- Фрагмент источника: глава 5, правило «JSON must be well-formed».
- Область применения: разработчик.

### RADR-065. XML и другие форматы — опционально

- Утверждение: XML, HTML и другие форматы допустимо поддерживать как
  альтернативные представления, выбираемые через согласование медиатипа.
- Фрагмент источника: глава 5, правило «XML and other formats may optionally be
  used».
- Область применения: разработчик.

### RADR-066. Запрет дополнительных «конвертов»

- Утверждение: дополнительные конверты (envelopes) создаваться не должны — тело
  должно содержать только представление состояния ресурса, опираясь на «конверт»
  самого HTTP.
- Фрагмент источника: глава 5, правило «Additional envelopes must not be
  created».
- Область применения: разработчик.

### RADR-067. Единообразная форма ссылок и связей

- Утверждение: следует использовать единообразную форму для представления ссылок
  (link) и связей (link relation), включаемых в представления вместе с полями.
- Фрагмент источника: глава 5, правила «A consistent form should be used to
  represent links / link relations».
- Область применения: архитектор.

### RADR-068. Обязательная ссылка self

- Утверждение: представление идентифицируемого ресурса должно включать ссылку с
  именем `self`, указывающую на эквивалентный самому ресурсу URI.
- Фрагмент источника: глава 5, правило «A self link should be included».
- Область применения: разработчик.

### RADR-069. Минимизация числа entry-point URI

- Утверждение: число рекламируемых «точек входа» следует минимизировать —
  публиковать URI docroot, чьё представление даёт ссылки на остальные ресурсы, а
  не список всех URI (иначе возникает тесная связанность клиентов).
- Фрагмент источника: глава 5, правило «Minimize the number of advertised entry
  point API URIs».
- Область применения: архитектор.

### RADR-070. Ссылки как контекстно-зависимое меню действий (HATEOAS)

- Утверждение: следует использовать ссылки для рекламы доступных действий
  ресурса с учётом его состояния; HATEOAS требует, чтобы каждый ответ содержал
  состояние-зависимые ссылки.
- Фрагмент источника: глава 5, правило «Links should be used to advertise a
  resource's available actions in a state-sensitive manner».
- Область применения: архитектор.

### RADR-071. Единообразная форма ошибки и контейнера ошибок

- Утверждение: следует использовать единообразную форму ошибки с полями `id`
  (код типа) и `description`, а ответ об ошибках оформлять как `ErrorContainer`
  со списком `elements` и статусом 4xx или 5xx.
- Фрагмент источника: глава 5, правила «A consistent form should be used to
  represent errors / error responses».
- Область применения: разработчик.

### RADR-072. Общие типы ошибок

- Утверждение: для типичных ошибок следует использовать согласованные общие типы
  ошибок, определённые один раз и переиспользуемые через расширение схем.
- Фрагмент источника: глава 5, правило «Consistent error types should be used».
- Область применения: архитектор.

## Глава 6. Client Concerns

### RADR-073. Новый URI для нового концепта (версионирование)

- Утверждение: версию API или ресурса, как правило, не следует обозначать в URI
  (например, `v2`); новый URI следует вводить только для нового концепта,
  сохраняя устойчивое сопоставление URI и ресурсов.
- Фрагмент источника: глава 6, правило «New URIs should be used to introduce new
  concepts».
- Область применения: архитектор.

### RADR-074. Версии формы — через схемы

- Утверждение: версии формы представлений следует вести через версионированные
  схемы; добавление полей и ссылок в новые версии схемы вводит новые возможности
  без потери обратной совместимости.
- Фрагмент источника: глава 6, правило «Schemas should be used to manage
  representational form versions».
- Область применения: архитектор.

### RADR-075. Версии состояния — через ETag

- Утверждение: для управления версиями состояния представления следует
  использовать entity tags (ETag) — наиболее тонкозернистую систему
  версионирования.
- Фрагмент источника: глава 6, правило «Entity tags should be used to manage
  representational state versions».
- Область применения: разработчик.

### RADR-076. OAuth для защиты ресурсов

- Утверждение: для защиты ресурсов допустимо использовать OAuth — открытый
  HTTP-протокол авторизации, дополняющий ресурсно-ориентированную и
  stateless-природу REST.
- Фрагмент источника: глава 6, правило «OAuth may be used to protect resources».
- Область применения: специалист по безопасности.

### RADR-077. API management для защиты ресурсов

- Утверждение: для защиты ресурсов допустимо использовать решения API management
  (reverse proxy), закрывающие сквозные задачи и поддерживающие OAuth «из
  коробки».
- Фрагмент источника: глава 6, правило «API management solutions may be used to
  protect resources».
- Область применения: специалист по безопасности.

### RADR-078. Частичные ответы через query-параметр fields

- Утверждение: для частичных ответов следует использовать query-параметр
  `fields` (список включения или, с `!`, исключения); API обязан разобрать
  список и вернуть частичное представление.
- Фрагмент источника: глава 6, правило «The query component should be used to
  support partial responses».
- Область применения: разработчик.

### RADR-079. Встраивание связанных ресурсов через query

- Утверждение: следует позволять клиенту через query управлять тем, какие
  связанные ресурсы остаются «обычными» ссылками, а какие становятся
  «встроенными» (embedded), для request-time композиции.
- Фрагмент источника: глава 6, правило «The query component should be used to
  embed linked resources».
- Область применения: разработчик.

### RADR-080. JSONP для кросс-доменного чтения

- Утверждение: следует поддерживать JSONP для многодоменного доступа на чтение
  из JavaScript (через query-параметр `callback`); из-за инъекции элемента
  `<script>` JSONP ограничен запросами GET.
- Фрагмент источника: глава 6, правило «JSONP should be supported».
- Область применения: разработчик.

### RADR-081. CORS для кросс-доменного чтения и записи

- Утверждение: следует поддерживать CORS как стандартизированную W3C
  альтернативу JSONP для многодоменного доступа на чтение и запись всеми
  методами; для методов, кроме GET/HEAD/POST, используется preflight-запрос, а
  разрешённые источники задаются заголовком `Access-Control-Allow-Origin`.
- Фрагмент источника: глава 6, правило «CORS should be supported».
- Область применения: специалист по безопасности.
