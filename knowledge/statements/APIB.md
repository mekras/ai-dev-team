## Назначение

Этот документ фиксирует набор утверждений по книге `APIB`, достаточный для того,
чтобы роли команды (архитектор, разработчик, аналитик, технический писатель,
менеджер проекта, специалист по безопасности) уверенно и правильно работали с
проектированием, развитием и продуктовым сопровождением API, а также для
проверки влияния на навыки и документы проекта.

Утверждения не являются требованиями, правилами или решениями для этого
репозитория. Они фиксируют наблюдаемое содержание книги.

Каждое утверждение опирается на локальную текстовую выгрузку
`knowledge/primary/APIB/files/the-api-book-ru.txt`, полученную из PDF утилитой
`pdftotext -layout`. Полные артефакты являются локальными неотслеживаемыми
данными. Все утверждения имеют статус `ready_for_review`. Главы в книге
пронумерованы; разделы внутри глав указаны по названиям. Дословные крупные
фрагменты не приводятся: содержание пересказано.

## Введение (главы 1–6)

### APIB-001. API как обязательство

- Утверждение: книга определяет API по смыслу как формальное обязательство
  связывать между собой различные программируемые контексты, а не просто как
  «программный интерфейс приложений».
- Фрагмент источника: глава 2 «Определение API».
- Область применения: архитектор.

### APIB-002. Пользователи API — инженеры

- Утверждение: потребителями API являются разработчики, поэтому поставщик не
  контролирует качество чужих инженерных решений, но проблемы будут приписывать
  ему.
- Фрагмент источника: глава 2 «Определение API».
- Область применения: менеджер проекта.

### APIB-003. API — мультипликатор ошибок

- Утверждение: API одновременно мультиплицирует и возможности, и ошибки
  разработчика, что налагает повышенную ответственность.
- Фрагмент источника: глава 2 «Определение API».
- Область применения: архитектор.

### APIB-004. Три критерия качества

- Утверждение: базовые критерии хорошего API — очевидность (как решить задачу),
  читаемость (правильный код по номенклатуре) и консистентность (действие по
  аналогии).
- Фрагмент источника: глава 3 «Критерии качества API».
- Область применения: разработчик.

### APIB-005. Главные проблемы — при развитии

- Утверждение: нелогичность и неконсистентность возникают не в начальном ядре
  API, а при его развитии и попытках сохранить совместимость.
- Фрагмент источника: глава 3 «Критерии качества API».
- Область применения: архитектор.

### APIB-006. Закладывание на будущее небесплатно

- Утверждение: проектирование «на будущее» означает избыточность интерфейсов и
  абстрагирования и оправдано, только если у API это будущее есть, иначе это
  оверинжиниринг.
- Фрагмент источника: глава 3 «Критерии качества API».
- Область применения: архитектор.

### APIB-007. Совместимость как временна́я характеристика

- Утверждение: обратная совместимость — временно́е свойство качества, которое
  отличает разработку API от разработки ПО вообще.
- Фрагмент источника: глава 4 «Обратная совместимость».
- Область применения: архитектор.

### APIB-008. Слом совместимости — скрытый налог

- Утверждение: выпуск обратно несовместимых версий — «налог» на потребителей; не
  следует ломать совместимость, если это возможно.
- Фрагмент источника: глава 4 «Обратная совместимость».
- Область применения: менеджер проекта.

### APIB-009. Принятие semver

- Утверждение: книга принимает версионирование semver и устанавливает
  эквивалентность «мажорная версия» = «обратно несовместимое изменение
  функциональности».
- Фрагмент источника: глава 5 «О версионировании».
- Область применения: разработчик.

### APIB-010. Нейтральный термин «сущность»

- Утверждение: книга намеренно избегает слов «метод», «объект», «функция» в
  пользу нейтрального термина «сущность» как атомарной единицы функциональности.
- Фрагмент источника: глава 6 «Условные обозначения и терминология».
- Область применения: технический писатель.

## Раздел I. Проектирование API (главы 7–12)

### APIB-011. Алгоритм проектирования из четырёх шагов

- Утверждение: проектирование ведётся сверху вниз из четырёх шагов: определение
  области применения, разделение уровней абстракции, разграничение областей
  ответственности, описание конечных интерфейсов.
- Фрагмент источника: глава 7 «Пирамида контекстов API».
- Область применения: архитектор.

### APIB-012. Цена ошибки растёт к вершине

- Утверждение: цена ошибки различна по уровням: плохое именование исправить
  просто, а неверное понимание назначения API — практически невозможно.
- Фрагмент источника: глава 7 «Пирамида контекстов API».
- Область применения: архитектор.

### APIB-013. Четырёхкратный вопрос о проблеме

- Утверждение: книга предлагает четыре раза задать вопрос «какую проблему мы
  решаем?» с ударением на каждом слове, чтобы проверить нужность, компетентность
  и адекватность решения.
- Фрагмент источника: глава 8 «Определение области применения».
- Область применения: аналитик.

### APIB-014. Цели уровней абстракции

- Утверждение: выделение уровней абстракции преследует три цели — упрощение
  работы разработчика, поддержку обратной совместимости и интероперабельность.
- Фрагмент источника: глава 9 «Разделение уровней абстракции».
- Область применения: архитектор.

### APIB-015. Смешение уровней через общее поле

- Утверждение: одно и то же поле, означающее разное в разных интерфейсах, —
  признак смешения уровней абстракции.
- Фрагмент источника: глава 9 «Разделение уровней абстракции».
- Область применения: разработчик.

### APIB-016. Изоляция уровней небесплатна

- Утверждение: соблюдение изоляции уровней усложняет интерфейс, но никакая
  гибкость и расширяемость не бывают бесплатными.
- Фрагмент источника: глава 9 «Разделение уровней абстракции».
- Область применения: архитектор.

### APIB-017. Дерево потоков данных

- Утверждение: дерево потоков данных — способ проверить разделение уровней и
  развивать большие API, которые невозможно удержать в памяти целиком.
- Фрагмент источника: глава 9 «Разделение уровней абстракции», раздел о потоках
  данных.
- Область применения: архитектор.

### APIB-018. Каждый уровень — своя предметная область

- Утверждение: каждый уровень иерархии — самостоятельная промежуточная
  предметная область, через которую задача переводится из одного контекста в
  другой.
- Фрагмент источника: глава 10 «Разграничение областей ответственности».
- Область применения: аналитик.

### APIB-019. Переформулирование операций по уровням

- Утверждение: модифицирующие операции (например, отмена заказа) на каждом
  уровне абстракции переформулируются и не должны «перепрыгивать» уровни.
- Фрагмент источника: глава 10 «Разграничение областей ответственности».
- Область применения: разработчик.

### APIB-020. Проверка через сценарии использования

- Утверждение: на этапе спроектированной номенклатуры следует встать на место
  разработчика и попробовать написать код, чтобы обнаружить неудобства API.
- Фрагмент источника: глава 10 «Разграничение областей ответственности», раздел
  о сценариях использования.
- Область применения: аналитик.

### APIB-021. Правила применять осмысленно

- Утверждение: «нулевое правило» гласит, что правила — обобщение опыта и не
  должны применяться бездумно, если нет причины им следовать.
- Фрагмент источника: глава 11 «Описание конечных интерфейсов», правило 0.
- Область применения: разработчик.

### APIB-022. Явное лучше неявного

- Утверждение: из названия сущности должно быть очевидно, что она делает и к
  каким побочным эффектам приводит; модифицирующих операций за GET быть не
  может.
- Фрагмент источника: глава 11 «Описание конечных интерфейсов», правило явности.
- Область применения: разработчик.

### APIB-023. Закон больших чисел против вас

- Утверждение: если сигнатуру можно понять неправильно, то по мере роста
  популярности API её неизбежно будет понимать неправильно всё больше партнёров.
- Фрагмент источника: глава 11 «Описание конечных интерфейсов», раздел о
  читабельности.
- Область применения: разработчик.

### APIB-024. Не экономить на именах

- Утверждение: сущности следует именовать конкретно, не экономить буквы и делать
  тип поля ясным из названия для лучшей читаемости.
- Фрагмент источника: глава 11 «Описание конечных интерфейсов», правила
  именования.
- Область применения: разработчик.

### APIB-025. Информативные ошибки

- Утверждение: ошибки должны быть информативными; из интерфейса должны быть
  понятны не только решение задачи, но и возможные ошибки и исключения.
- Фрагмент источника: глава 11 «Описание конечных интерфейсов», правило об
  ошибках.
- Область применения: разработчик.

### APIB-026. Идемпотентность операций

- Утверждение: книга формулирует правило об идемпотентности операций и
  предостерегает от неатомарных операций.
- Фрагмент источника: глава 11 «Описание конечных интерфейсов», правила об
  идемпотентности и атомарности.
- Область применения: разработчик.

### APIB-027. Не изобретать безопасность

- Утверждение: не следует изобретать собственную безопасность; технические
  ограничения API нужно явно декларировать.
- Фрагмент источника: глава 11 «Описание конечных интерфейсов», правила о
  безопасности.
- Область применения: специалист по безопасности.

### APIB-028. Локализация продуктовых данных

- Утверждение: локализация и интернационализация продуктовых данных вынесены в
  отдельное правило проектирования интерфейсов.
- Фрагмент источника: глава 11 «Описание конечных интерфейсов», правило о
  локализации.
- Область применения: технический писатель.

## Раздел II. Обратная совместимость (главы 13–19)

### APIB-029. Совместимость = функциональная стабильность

- Утверждение: обратная совместимость — свойство системы оставаться
  функционально корректной во времени, то есть код потребителей продолжает
  решать свою задачу.
- Фрагмент источника: глава 13 «Постановка проблемы обратной совместимости».
- Область применения: архитектор.

### APIB-030. Длительность увязана с жизненным циклом платформ

- Утверждение: срок поддержки совместимости следует увязывать с жизненными
  циклами приложений и LTS-периодами платформ (обычно несколько лет).
- Фрагмент источника: глава 13 «Постановка проблемы обратной совместимости».
- Область применения: менеджер проекта.

### APIB-031. Три процесса размывания

- Утверждение: совместимость подрывают три процесса: фрагментация клиентов,
  размывание предметной области и нижележащей платформы.
- Фрагмент источника: глава 13 «Постановка проблемы обратной совместимости».
- Область применения: архитектор.

### APIB-032. Правило айсберга 1:10

- Утверждение: в хорошем API документированная и недокументированная части
  соотносятся примерно как 1 к 10, и наружу нужно выставлять минимум
  функциональности.
- Фрагмент источника: глава 14 «О ватерлинии айсберга».
- Область применения: архитектор.

### APIB-033. Каждая публичная сущность — продуктовое решение

- Утверждение: каждое публичное поле, метод и сущность — продуктовое решение,
  которое должно иметь веские причины для документирования.
- Фрагмент источника: глава 14 «О ватерлинии айсберга».
- Область применения: менеджер проекта.

### APIB-034. Нельзя принять обязательства наполовину

- Утверждение: не следует легитимизировать серые зоны (недокументированные поля,
  использование непубличных особенностей в примерах): либо функциональность
  гарантируется всегда, либо на её существование не намекают.
- Фрагмент источника: глава 14 «О ватерлинии айсберга».
- Область применения: технический писатель.

### APIB-035. Фиксировать неявные договорённости

- Утверждение: неявные договорённости (например, о модели консистентности) нужно
  явно фиксировать в документации.
- Фрагмент источника: глава 14 «О ватерлинии айсберга».
- Область применения: разработчик.

### APIB-036. Расширение через абстрагирование

- Утверждение: функциональность следует расширять так, чтобы существующие методы
  становились частным упрощённым случаем более общего интерфейса.
- Фрагмент источника: глава 15 «Расширение через абстрагирование»; глава 18
  «Интерфейсы как универсальный паттерн».
- Область применения: архитектор.

### APIB-037. Сильная связность схлопывает интерфейсы

- Утверждение: попытка модифицировать базовую функциональность (например,
  авторские рецепты партнёров) порождает проблемы сильной связности.
- Фрагмент источника: глава 16 «Сильная связность и сопутствующие проблемы».
- Область применения: архитектор.

### APIB-038. Вышестоящие сущности — информационные контексты

- Утверждение: вышестоящие сущности должны быть информационными контекстами для
  нижестоящих: сообщать о состоянии и давать средства его менять, а не
  предписывать поведение.
- Фрагмент источника: глава 18 «Интерфейсы как универсальный паттерн».
- Область применения: архитектор.

### APIB-039. Связь с принципами SOLID

- Утверждение: правила расширения узнаются как принципы SOLID, поскольку API по
  определению является контрактом; книга добавляет к ним понятие уровней
  абстракции.
- Фрагмент источника: глава 18 «Интерфейсы как универсальный паттерн».
- Область применения: архитектор.

### APIB-040. Сущность как реализация общего интерфейса

- Утверждение: каждую сущность стоит изначально рассматривать как частную
  реализацию более общего интерфейса, даже если альтернатив пока не предвидится.
- Фрагмент источника: глава 18 «Интерфейсы как универсальный паттерн».
- Область применения: разработчик.

### APIB-041. Даже исправление ошибки ломает код

- Утверждение: даже исправление ошибки может сломать чужой код, и иногда
  приходится эмулировать ошибочное поведение до мажорного релиза.
- Фрагмент источника: глава 19 «Блокнот душевного покоя».
- Область применения: разработчик.

### APIB-042. Тестировать формальные интерфейсы

- Утверждение: тестировать нужно именно формальные интерфейсы API, включая
  «самоочевидные» инварианты и проверку абстрактных интерфейсов отдельной
  реализацией.
- Фрагмент источника: глава 19 «Блокнот душевного покоя».
- Область применения: разработчик.

### APIB-043. Изолировать сторонние зависимости

- Утверждение: не следует проксировать чужой API «как есть»; сторонние
  зависимости нужно изолировать программной обвязкой с точками расширения.
- Фрагмент источника: глава 19 «Блокнот душевного покоя».
- Область применения: архитектор.

## Раздел III. API как продукт (главы 20–32)

### APIB-044. API — полноценный, но специфический продукт

- Утверждение: API — полноценный продукт, к которому применимы принципы
  продакт-менеджмента, и одновременно очень специфический продукт.
- Фрагмент источника: глава 20 «Продукт API».
- Область применения: менеджер проекта.

### APIB-045. Пирамида аудитории API

- Утверждение: аудитория API образует пирамиду: пользователи (основание),
  бизнес-заказчики (середина) и разработчики (вершина), интересы которых не
  всегда совпадают.
- Фрагмент источника: глава 20 «Продукт API».
- Область применения: менеджер проекта.

### APIB-046. Главная проблема развития — отсутствие API-экспертизы

- Утверждение: самая большая проблема развития API — отсутствие у руководителей
  продукта и маркетологов экспертизы в работе с API.
- Фрагмент источника: глава 20 «Продукт API».
- Область применения: менеджер проекта.

### APIB-047. Модели монетизации API

- Утверждение: книга разбирает основные модели монетизации (платные лицензии,
  открытая лицензия с ограничениями, платная поддержка, спонсирование владельцем
  платформы, тарификация доступа к сервису).
- Фрагмент источника: глава 21 «Бизнес-модели API».
- Область применения: менеджер проекта.

### APIB-048. Недоступность дешёвого MVP

- Утверждение: для API недоступен дешёвый MVP, так как для проверки гипотезы
  партнёры должны вложить деньги в написание кода; решением служат компенсации
  или референсное приложение.
- Фрагмент источника: глава 22 «Формирование продуктового видения».
- Область применения: менеджер проекта.

### APIB-049. Рациональность аудитории разработчиков

- Утверждение: разработчики — рациональная аудитория, скептичная к рекламе и
  мнениям инфлюэнсеров, требующая цифр и примеров кода.
- Фрагмент источника: глава 23 «Взаимодействие с разработчиками».
- Область применения: менеджер проекта.

### APIB-050. Специфические каналы технопиара

- Утверждение: до разработчиков информацию доносят через специфические каналы
  (коллективные блоги, Q&A-сайты, обучающие сервисы, конференции), а прямая
  реклама там малоэффективна.
- Фрагмент источника: глава 23 «Взаимодействие с разработчиками».
- Область применения: менеджер проекта.

### APIB-051. Две категории потребителей

- Утверждение: потребители делятся на профессиональных и
  начинающих/полупрофессиональных разработчиков; один API не может одинаково
  хорошо подходить обоим.
- Фрагмент источника: глава 23 «Взаимодействие с разработчиками».
- Область применения: технический писатель.

### APIB-052. Парадокс работы с бизнесом

- Утверждение: бизнес лояльнее к предложениям, особенно бесплатным, но ему
  гораздо сложнее объяснить преимущества API как концепции.
- Фрагмент источника: глава 24 «Взаимодействие с бизнес-аудиторией».
- Область применения: менеджер проекта.

### APIB-053. Нужна линейка, а не один API

- Утверждение: нет смысла поставлять один API; следует говорить о наборе
  продуктов сразу в двух измерениях (горизонтальном и вертикальном).
- Фрагмент источника: глава 25 «Линейка сервисов API».
- Область применения: менеджер проекта.

### APIB-054. KPI — целевые действия, а не сырые цифры

- Утверждение: сырое число пользователей и партнёров — необходимый минимум, но
  не KPI; KPI следует выставлять по количеству целевых действий, коррелирующих с
  монетизацией.
- Фрагмент источника: глава 26 «Ключевые показатели эффективности API».
- Область применения: аналитик.

### APIB-055. Запас прочности API выше обычного

- Утверждение: нагрузку на API сложно предсказать (одна строка кода партнёра
  может её резко повысить), поэтому запас прочности API должен быть существенно
  выше, чем у пользовательских сервисов.
- Фрагмент источника: глава 26 «Ключевые показатели эффективности API», раздел о
  SLA.
- Область применения: архитектор.

### APIB-056. Уязвимость API тиражируется на потребителей

- Утверждение: уязвимость в API фактически создаёт дыру в безопасности в каждом
  приложении партнёра, поэтому требуются максимальные меры защиты и оперативная
  реакция.
- Фрагмент источника: глава 26 «Ключевые показатели эффективности API», раздел о
  SLA.
- Область применения: специалист по безопасности.

### APIB-057. Ненадёжность API-ключей

- Утверждение: API-ключи не позволяют надёжно идентифицировать ни приложение, ни
  владельца и не освобождают от перепроверки соблюдения правил.
- Фрагмент источника: глава 27 «Идентификация пользователей и борьба с фродом».
- Область применения: специалист по безопасности.

### APIB-058. Подсчёт по IP и подсетям

- Утверждение: рекомендуется вести иерархический подсчёт уникальных IP (по
  подсетям /8, /16, /24, автономным системам) и мониторить публичные прокси и
  TOR, особенно после распространения IPv6.
- Фрагмент источника: глава 27 «Идентификация пользователей и борьба с фродом».
- Область применения: специалист по безопасности.

### APIB-059. Три шага антифрода

- Утверждение: отсеивание нежелательных запросов состоит из трёх шагов:
  идентификация подозрительных пользователей, запрос дополнительного фактора
  аутентификации, вынесение и применение решения об ограничении.
- Фрагмент источника: глава 28 «Технические способы борьбы с несанкционированным
  доступом к API».
- Область применения: специалист по безопасности.

### APIB-060. Скудость средств ограничения доступа

- Утверждение: реальных средств бана немного (по IP/подсети, обязательная
  идентификация, ложный ответ, административные методы), а бан по
  cookie/Referer/User-Agent почти не работает, так как эти данные подменяет
  клиент.
- Фрагмент источника: глава 28 «Технические способы борьбы с несанкционированным
  доступом к API».
- Область применения: специалист по безопасности.

### APIB-061. Высокая планка к специалистам поддержки

- Утверждение: поддержка API требует разработчиков, поскольку причину проблемы в
  чужом коде может понять только программист; при этом большинство запросов
  исходят от новичков и не касаются работы API.
- Фрагмент источника: глава 29 «Поддержка пользователей API».
- Область применения: менеджер проекта.

### APIB-062. Время на поддержку нужно планировать

- Утверждение: время на техподдержку нужно закладывать при планировании, так как
  чтение и удалённая отладка незнакомого кода крайне трудозатратны и растут с
  числом платформ и функциональности.
- Фрагмент источника: глава 29 «Поддержка пользователей API».
- Область применения: менеджер проекта.

### APIB-063. Документация — лицо продукта

- Утверждение: документация — лицо продукта и точка входа в него; пользователи
  работают со справкой не так, как представляет автор API.
- Фрагмент источника: глава 30 «Документация».
- Область применения: технический писатель.

### APIB-064. Спецификация — не документация

- Утверждение: формальная спецификация (в том числе машиночитаемая, например
  OpenAPI) сама по себе не является документацией; документация — это словесные
  описания в поле description.
- Фрагмент источника: глава 30 «Документация».
- Область применения: технический писатель.

### APIB-065. Примеры кода — главный инструмент

- Утверждение: примеры кода — важнейший инструмент привлечения новых
  пользователей; они должны быть лаконичными, атомарными и близкими к реальному
  приложению.
- Фрагмент источника: глава 30 «Документация».
- Область применения: технический писатель.

### APIB-066. Quick start как индикатор

- Утверждение: quick start длиной более десятка строк кода — признак того, что в
  определении частотных кейсов и хелперных методов что-то сделано не так.
- Фрагмент источника: глава 30 «Документация», раздел о руководстве.
- Область применения: технический писатель.

### APIB-067. Необходимость тестовой среды

- Утверждение: для API, чьи операции имеют последствия для пользователей (в том
  числе стоят денег), необходима тестовая версия с симуляцией реальных действий.
- Фрагмент источника: глава 31 «Тестовая среда».
- Область применения: разработчик.

### APIB-068. Два подхода к тестовой среде

- Утверждение: книга сравнивает два подхода к тестовой среде — мета-API среды
  тестирования (хелперные методы) и симулятор предопределённых сценариев; у
  каждого свои плюсы и минусы.
- Фрагмент источника: глава 31 «Тестовая среда».
- Область применения: разработчик.

### APIB-069. Цель тестовой среды — автоматизация

- Утверждение: конечная цель тестового API — позволить партнёрам
  автоматизировать тестирование, в том числе симулировать сценарии (например,
  3-D Secure) в ускоренном масштабе времени.
- Фрагмент источника: глава 31 «Тестовая среда».
- Область применения: разработчик.

### APIB-070. Золотое правило выпуска версий

- Утверждение: период, после которого партнёру потребуется переписать код,
  должен совпадать с жизненным циклом приложений в предметной области.
- Фрагмент источника: глава 32 «Управление ожиданиями».
- Область применения: менеджер проекта.

### APIB-071. Публичные роадмапы как преимущество

- Утверждение: следует публиковать (пусть условные, без дат) роадмапы, особенно
  при прекращении поддержки функциональности; это важное конкурентное
  преимущество.
- Фрагмент источника: глава 32 «Управление ожиданиями».
- Область применения: менеджер проекта.
