## Существенные сведения источников

Извлечено `71` утверждение по `APIB` (книга Сергея Константинова «API») и `81`
утверждение по `RADR` (книга Mark Massé «REST API Design Rulebook»). Источники
взаимодополняют друг друга: `APIB` даёт методологию проектирования и эволюции
API (сверху вниз, обратная совместимость, API как продукт), `RADR` — конкретный
свод правил веб-уровня (URI, HTTP, метаданные, представления, гипермедиа).

Значимые группы:

- `APIB-011`–`APIB-028` — проектирование сверху вниз: область применения, уровни
  абстракции, разграничение ответственности, правила конечных интерфейсов.
- `APIB-029`–`APIB-043` — обратная совместимость: функциональная стабильность,
  слабая и сильная связность, расширение через абстрагирование, изоляция
  зависимостей.
- `APIB-044`–`APIB-071` — API как продукт: аудитория, бизнес-модели, KPI,
  безопасность и фрод, поддержка, документация, тестовая среда, версии.
- `RADR-001`–`RADR-022` — архитектура Веба, модель зрелости Ричардсона, WRML,
  правила URI и архетипы ресурсов.
- `RADR-023`–`RADR-050` — назначение HTTP-методов и корректный выбор кодов
  состояния 2xx/3xx/4xx/5xx.
- `RADR-051`–`RADR-072` — метаданные (заголовки, кэширование, медиатипы) и
  представления (JSON, гипермедиа, формы ссылок и ошибок).
- `RADR-073`–`RADR-081` — потребности клиентов: версионирование, безопасность,
  частичные ответы, JSONP/CORS.

## Находки

### Severity: Medium — проектирование API как отдельная стилезависимая дисциплина

Документ: новая справка
`.apm/skills/software-architecture/references/api-design.md`

Источники: `APIB`, `RADR`, проверка `2026-06-24`

Первичные данные: `APIB-001`–`APIB-043`, `APIB-063`–`APIB-066`,
`RADR-001`–`RADR-081`

Класс: новое существенное сведение (стилезависимое)

Факт источников: вместе книги дают цельную дисциплину проектирования API:
методологию «сверху вниз» (область применения → уровни абстракции → области
ответственности → конечные интерфейсы), критерии качества (очевидность,
читаемость, консистентность), дисциплину обратной совместимости (слабая
связность, расширение через абстрагирование, «ватерлиния айсберга», изоляция
зависимостей) и конкретный свод правил веб-уровня (архетипы ресурсов и URI,
назначение HTTP-методов и кодов состояния, метаданные и кэширование,
гипермедиа/HATEOAS, единообразные формы ссылок и ошибок, версионирование,
безопасность, CORS).

Область применения: задачи проектирования и эволюции прикладных и веб-API; не
выбор по умолчанию для любой системы.

Вывод: в навыке `software-architecture` есть общее понятие контракта и
публичного интерфейса (`component-design.md`, `quality-criteria.md`), но нет
отдельной справки по проектированию API. Материал цельный и ценный, но привязан
к классу задач (проектирование API). По аналогии с `clean-architecture.md`,
`distributed-tradeoffs.md` и `evolutionary-architecture.md` уместно вынести его
в отдельную справку, помеченную как стилезависимую. Изменяет состав навыка →
**требует решения владельца**.

Статус переноса: `apply_now` (владелец одобрил `2026-06-24`)

Что сделать: создать `api-design.md` (методология «сверху вниз», критерии
качества, обратная совместимость и связность, правила веб-уровня по `RADR`, API
как продукт кратко со ссылкой на документацию), пометить как стилезависимую и
добавить ссылку в `SKILL.md` рядом с `evolutionary-architecture.md`.
**Выполнено.**

### Severity: Low — частичное усиление существующих разделов о контрактах

Документ: `.apm/skills/software-architecture/references/component-design.md`,
`.apm/skills/software-architecture/references/quality-criteria.md`

Источники: `APIB`, проверка `2026-06-24`

Первичные данные: `APIB-007`–`APIB-009`, `APIB-029`–`APIB-040`

Класс: частично покрыто

Факт источников: `component-design.md` уже требует устойчивого публичного
контракта, обратной совместимости и проверок совместимости;
`quality-criteria.md` содержит совместимость и версионирование. `APIB` усиливает
это пониманием совместимости как временно́й характеристики качества,
эквивалентностью «мажорная версия = несовместимое изменение» и приёмами слабой
связности и расширения через абстрагирование.

Область применения: ядро навыка о контрактах и совместимости.

Вывод: тема в основном покрыта; новизна — формулировки о временно́й природе
совместимости и приёмах эволюции контракта. Уместно дать их в составе новой
справки `api-design.md` со ссылками на существующие разделы, не дублируя их.

Статус переноса: `applied` (в составе новой справки)

Что сделано: в `api-design.md` даны ссылки на `component-design.md` и
`quality-criteria.md`, добавлена только специфика API; существующие разделы не
дублируются.

### Severity: Low — документация API как возможное усиление technical-writing

Документ: навык `technical-writing`

Источники: `APIB`, проверка `2026-06-24`

Первичные данные: `APIB-063`–`APIB-066`

Класс: новое сведение (узкое)

Факт источников: документация — лицо продукта; формальная спецификация (в том
числе OpenAPI) не равна документации; примеры кода — главный инструмент
привлечения; разросшийся quick start — признак недоработки частотных сценариев.

Область применения: подготовка документации API.

Вывод: это узкое и стилезависимое сведение о документировании API. Отдельная
правка навыка `technical-writing` не очевидна; целесообразнее упомянуть в
разделе «API как продукт» новой справки `api-design.md` с указанием связи с
`technical-writing`. Самостоятельным основанием для изменения
`technical-writing` не является.

Статус переноса: `no_change` (фиксируется как сведение; правка навыка не
предлагается)
