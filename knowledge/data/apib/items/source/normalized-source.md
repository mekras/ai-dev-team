## Структура книги

Книга проводит читателя по жизненному циклу API — от замысла до сопровождения
зрелого продукта — и держит сквозную мысль: API есть формальное обязательство
связывать программируемые контексты, поэтому ошибки проектирования и нарушение
совместимости умножаются на число потребителей. Изложение строится на одном
сквозном учебном примере (API заказа кофе) и движется сверху вниз: смысл и
область применения, затем уровни абстракции и интерфейсы, затем эволюция и
совместимость и, наконец, продуктовые и нетехнические аспекты. Автор считает
третий, продуктовый, раздел самым важным.

### Введение (главы 1–6)

- Глава 1 «О структуре этой книги»: три раздела (проектирование, совместимость,
  продукт) и тезис о первичности продуктового раздела.
- Глава 2 «Определение API»: смысловое определение API как формального
  обязательства связывать программируемые контексты; потребители API — инженеры.
- Глава 3 «Критерии качества API»: очевидность, читаемость, консистентность;
  предупреждение об оверинжиниринге.
- Глава 4 «Обратная совместимость»: совместимость как временна́я характеристика
  качества и «скрытый налог» на потребителей при сломе совместимости.
- Глава 5 «О версионировании»: semver и эквивалентность «мажорная версия» =
  «обратно несовместимое изменение».
- Глава 6 «Условные обозначения и терминология»: нейтральный термин «сущность»,
  нотация JSON-over-HTTP и C-подобный псевдокод примеров.

### Раздел I. Проектирование API (главы 7–12)

- Глава 7 «Пирамида контекстов API»: алгоритм проектирования из четырёх шагов
  сверху вниз.
- Глава 8 «Определение области применения»: четырёхкратный вопрос «какую
  проблему мы решаем?» с ударением на каждом слове.
- Глава 9 «Разделение уровней абстракции»: цели изоляции уровней и инструмент
  «дерево потоков данных».
- Глава 10 «Разграничение областей ответственности»: формулирование назначения
  сущностей и проверка номенклатуры через сценарии использования.
- Глава 11 «Описание конечных интерфейсов»: набор практических правил именования
  и проектирования эндпойнтов (явность, идемпотентность, информативные ошибки,
  пагинация, безопасность, локализация).
- Глава 12 «Приложение к разделу I. Модельный API»: итоговый учебный API
  раздела.

### Раздел II. Обратная совместимость (главы 13–19)

- Глава 13 «Постановка проблемы обратной совместимости»: совместимость как
  функциональная стабильность во времени; три процесса её размывания.
- Глава 14 «О ватерлинии айсберга»: гигиенический минимум публичной поверхности
  API и фиксация неявных договорённостей.
- Глава 15 «Расширение через абстрагирование»: вариационный анализ «что, если?»
  и расширение через подключение реализаций.
- Глава 16 «Сильная связность и сопутствующие проблемы»: как сильная связность
  разрушает чистоту интерфейсов.
- Глава 17 «Слабая связность»: параметризация взаимодействия для разрыва жёстких
  зависимостей.
- Глава 18 «Интерфейсы как универсальный паттерн»: обобщение приёмов расширения;
  связь с принципами SOLID.
- Глава 19 «Блокнот душевного покоя»: конкретные рекомендации по совместимым
  изменениям, тестирование формальных интерфейсов, изоляция зависимостей.

### Раздел III. API как продукт (главы 20–32)

- Глава 20 «Продукт API»: API как полноценный, но специфический продукт;
  пирамида аудитории (пользователи — бизнес — разработчики).
- Глава 21 «Бизнес-модели API»: прямые и косвенные модели монетизации.
- Глава 22 «Формирование продуктового видения»: тройная аудитория и
  недоступность дешёвого MVP для API.
- Глава 23 «Взаимодействие с разработчиками»: рациональная аудитория, каналы
  технопиара, Open Source, профи и новички.
- Глава 24 «Взаимодействие с бизнес-аудиторией»: лояльность бизнеса и трудность
  объяснить ценность API.
- Глава 25 «Линейка сервисов API»: набор продуктов в двух измерениях
  (горизонтальном и вертикальном).
- Глава 26 «Ключевые показатели эффективности API»: KPI по целевым действиям,
  SLA и повышенный запас прочности.
- Глава 27 «Идентификация пользователей и борьба с фродом»: идентификация
  приложений и владельцев, ненадёжность API-ключей, подсчёт по IP.
- Глава 28 «Технические способы борьбы с несанкционированным доступом»: три шага
  антифрода и скудость средств бана.
- Глава 29 «Поддержка пользователей API»: высокая планка к специалистам,
  доминирование вопросов новичков, роль комьюнити.
- Глава 30 «Документация»: документация как лицо продукта; виды документации.
- Глава 31 «Тестовая среда»: необходимость тестового API и два подхода к нему.
- Глава 32 «Управление ожиданиями»: золотое правило версий, поддержка платформ,
  публикация роадмапов.

## Связь с проектом

Нормализованный слой служит картой для извлечения утверждений
(`knowledge/statements/APIB.md`) и для анализа влияния на навыки проекта,
связанные с архитектурой, требованиями, технической документацией и управлением
проектом.
