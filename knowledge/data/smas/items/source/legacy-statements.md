## Назначение

Этот документ фиксирует набор утверждений по отчёту `SMAS`, достаточный для
того, чтобы роли команды (прежде всего архитектор и специалист по безопасности,
а также разработчик) уверенно и правильно работали с управлением доступом к API
в микросервисной и распределённой архитектуре, а также для проверки влияния на
навыки и роли проекта.

Утверждения не являются требованиями, правилами или решениями для этого
репозитория. Они фиксируют наблюдаемое содержание отчёта. Каждое утверждение
опирается на локальную текстовую выгрузку
`knowledge/primary/SMAS/files/securing-microservice-apis.txt`, полученную из PDF
утилитой `pdftotext -layout`. Полные артефакты являются локальными
неотслеживаемыми данными. Все утверждения имеют статус `ready_for_review`.
Английские технические термины сохранены как есть. Крупные дословные фрагменты
не приводятся: содержание пересказано.

## Глава 1. Microservice Architecture

### SMAS-001. Характеристики микросервиса

- Утверждение: микросервис обычно реализует одну функциональную ответственность
  и обладает свойствами независимой развёртываемости и управляемости, а также
  эфемерности и эластичности (экземпляры короткоживущие, число масштабируется
  под нагрузку).
- Фрагмент источника: глава 1, вводный раздел.
- Область применения: архитектор, специалист по безопасности.

### SMAS-002. Типичные технологии микросервисов влияют на безопасность

- Утверждение: типичные для микросервисов веб-API (HTTP/JSON) и контейнерные
  развёртывания должны учитываться в любом решении по контролю доступа к API.
- Фрагмент источника: глава 1, вводный раздел.
- Область применения: архитектор, специалист по безопасности.

### SMAS-003. Базовый словарь API

- Утверждение: отчёт фиксирует понятия — сервис, экземпляр сервиса,
  интерфейс/веб-API, эндпойнт API, запрос и ответ API, потребитель и поставщик
  API, посредник API (шлюз, сервисный прокси); один сервис может быть и
  потребителем, и поставщиком.
- Фрагмент источника: глава 1, раздел «The Microservice API Landscape».
- Область применения: архитектор, специалист по безопасности.

### SMAS-004. Рамка управления доступом IAAA

- Утверждение: контроль доступа к веб-API описывается рамкой `IAAA` —
  идентификация, аутентификация, авторизация и учёт/аудит (accountability).
- Фрагмент источника: глава 1, раздел «API Access Control for Microservices».
- Область применения: архитектор, специалист по безопасности.

### SMAS-005. Множественные идентичности в одном запросе

- Утверждение: сообщения API могут запускаться действиями пользователя или
  автоматическими событиями и проходить через посредников, поэтому участники
  должны уметь передавать и принимать в запросе несколько идентичностей с их
  атрибутами.
- Фрагмент источника: глава 1, подраздел «Identification».
- Область применения: специалист по безопасности, архитектор.

### SMAS-006. Множественные решения об авторизации на пути запроса

- Утверждение: в распределённой микросервисной архитектуре один запрос может
  пройти через несколько решений об авторизации по мере передачи от компонента к
  компоненту; любой аутентифицировавший идентичность компонент может принять
  решение об авторизации.
- Фрагмент источника: глава 1, подраздел «Authorization».
- Область применения: архитектор, специалист по безопасности.

### SMAS-007. Ценность аудита для расследования инцидентов

- Утверждение: важно фиксировать активность, вызванную запросами API, с
  максимально полным контекстом сообщения, чтобы иметь данные для расследования
  намеренных и непреднамеренных нарушений.
- Фрагмент источника: глава 1, подраздел «Accountability».
- Область применения: специалист по безопасности.

### SMAS-008. Нефункциональные качества решения доступа

- Утверждение: решение по контролю доступа в микросервисной архитектуре должно
  удовлетворять нефункциональным качествам — управляемости/операбельности
  (машинные интерфейсы для автоматизации), производительности (минимум
  добавленной задержки) и удобству для разработчика.
- Фрагмент источника: глава 1, раздел «Microservice Architecture Qualities».
- Область применения: архитектор, специалист по безопасности.

## Глава 2. Access Control for Microservices

### SMAS-009. Вся безопасность основана на доверии

- Утверждение: безопасность строится на доверии, у которого нет точной меры;
  хорошая архитектура безопасности прозрачна относительно того, куда вложено
  доверие, и представляет собой настройку баланса доверия и принятия риска.
- Фрагмент источника: глава 2, раздел «Establishing Trust».
- Область применения: архитектор, специалист по безопасности.

### SMAS-010. Сетевая сегментация и зоны

- Утверждение: сетевая сегментация — базовый механизм; сегменты следует держать
  как можно меньше, а пересечение границы зоны должно требовать более строгой
  проверки (например, валидации токена безопасности).
- Фрагмент источника: глава 2, раздел «Network-Level Controls».
- Область применения: архитектор, специалист по безопасности.

### SMAS-011. SSL/TLS для всех API

- Утверждение: SSL/TLS даёт конфиденциальность и целостность данных в передаче,
  аутентификацию сервера и опционально клиента; на современных CPU влияние на
  производительность незначительно, поэтому все API должны использовать SSL/TLS
  повсеместно.
- Фрагмент источника: глава 2, раздел «SSL/TLS».
- Область применения: архитектор, специалист по безопасности, разработчик.

### SMAS-012. SPIFFE для идентичности сервисов

- Утверждение: SPIFFE упрощает аутентификацию сервисов и выдаёт `SVID` —
  сертификаты, уникально идентифицирующие компоненты в микросервисной
  инфраструктуре, что полезно в динамичной среде с традиционно сложным
  управлением PKI.
- Фрагмент источника: глава 2, врезка «SPIFFE».
- Область применения: специалист по безопасности, архитектор.

### SMAS-013. Прикладной контроль через обмен токенами

- Утверждение: прикладные сущности устанавливают доверие через обмен токенами
  безопасности; доверенная третья сторона выпускает токены и применяет
  криптографию, позволяя сторонам без предварительных отношений установить
  доверие.
- Фрагмент источника: глава 2, раздел «Application-Level Controls».
- Область применения: архитектор, специалист по безопасности.

### SMAS-014. Ограничения веб-сессий и угон сессии

- Утверждение: идентификатор сессии привязан к акту аутентификации и служит
  заменой первичных факторов пользователя, поэтому угон сессии (session
  hijacking) даёт атакующему все права валидных учётных данных.
- Фрагмент источника: глава 2, раздел «The Problem with Traditional Web Tokens».
- Область применения: специалист по безопасности, разработчик.

### SMAS-015. Вклад SAML: PDP и PEP

- Утверждение: SAML ввёл важные паттерны контроля доступа — разделение клиентов,
  защищаемых ресурсов и провайдеров идентичности и различие между точкой
  принятия решения (PDP) и точкой его применения (PEP), которые могут быть
  централизованы или распределены.
- Фрагмент источника: глава 2, раздел «The Problem with Traditional Web Tokens».
- Область применения: архитектор, специалист по безопасности.

### SMAS-016. SAML не подходит для API и микросервисов

- Утверждение: SAML сложен, опирается на централизованное администрирование
  доверия и XML/SOAP и не помогает пользователю делегировать авторизацию между
  приложениями, поэтому он плохо подходит для API и микросервисов.
- Фрагмент источника: глава 2, раздел «The Problem with Traditional Web Tokens».
- Область применения: архитектор, специалист по безопасности.

### SMAS-017. Принцип недоверия приложениям

- Утверждение: современная модель токенов исходит из того, что клиентскому или
  серверному приложению нельзя доверять первичный фактор аутентификации
  (пароль); следует выдавать короткоживущие токены с ограниченными
  возможностями.
- Фрагмент источника: глава 2, раздел «Modern Tokens For APIs».
- Область применения: специалист по безопасности, архитектор.

### SMAS-018. API-ключи идентифицируют приложение, а не пользователя

- Утверждение: API-ключ — непрозрачный токен, идентифицирующий приложение, а не
  пользователя; он встроен в бинарный образ, может быть извлечён атакующим и
  поэтому не должен считаться авторитетным и заменять аутентификацию
  пользователя.
- Фрагмент источника: глава 2, раздел «API keys» и врезка «API Keys or OAuth».
- Область применения: специалист по безопасности, разработчик, архитектор.

### SMAS-019. OAuth 2.0 — протокол авторизации, не аутентификации

- Утверждение: OAuth 2.0 — предпочтительный фреймворк делегированной
  авторизации; это не протокол аутентификации, а способ для владельца ресурса
  дать приложению ограниченный по времени и области доступ к ресурсу от своего
  имени.
- Фрагмент источника: глава 2, раздел «OAuth 2.0».
- Область применения: архитектор, специалист по безопасности, разработчик.

### SMAS-020. OpenID Connect — слой аутентификации поверх OAuth

- Утверждение: OpenID Connect добавляет к OAuth аутентификацию: через доверенный
  канал (например, нативный браузер) пользователь аутентифицируется у провайдера
  идентичности, а приложению выдаётся ограниченный по области и времени
  ID-токен, что выводит приложения из «бизнеса аутентификации».
- Фрагмент источника: глава 2, раздел «OpenID Connect».
- Область применения: архитектор, специалист по безопасности.

### SMAS-021. Непрозрачные и прозрачные токены

- Утверждение: непрозрачный (by-reference) токен индексирует состояние на
  централизованном сервере и требует валидации у эмитента (простое отзывание, но
  узкое место); прозрачный (by-value) токен интерпретируется локально в точке
  принятия решения и лучше масштабируется; ID-токен OIDC — прозрачный.
- Фрагмент источника: глава 2, раздел «Opaque tokens versus transparent tokens».
- Область применения: архитектор, специалист по безопасности.

### SMAS-022. JWT и ABAC

- Утверждение: JWT — JSON-формат для обмена утверждениями (claims) с заголовком,
  телом, подписью (JWS) и опциональным шифрованием (JWE); утверждения позволяют
  применять локальную политику и лежат в основе ABAC — авторизации по атрибутам,
  а не по индивидуальной идентичности.
- Фрагмент источника: глава 2, раздел «JWT».
- Область применения: архитектор, специалист по безопасности.

### SMAS-023. Осторожность с JWT для сессий и стеком JOSE

- Утверждение: отчёт рекомендует использовать JWT для авторизации, но избегать
  его для управления пользовательскими сессиями, и отмечает выявленные проблемы
  стека JOSE (ранние реализации позволяли подделывать токены, нет механизма
  версионирования для исключения слабых алгоритмов).
- Фрагмент источника: глава 2, врезка «JWT Sessions and JOSE Issues».
- Область применения: специалист по безопасности, разработчик.

### SMAS-024. Различие прокси и API-шлюза

- Утверждение: прокси и шлюзы — обратные прокси между клиентом и сервером; шлюзы
  работают на более высоком уровне, программируемы и обычно отвечают за
  аутентификацию, авторизацию, обнаружение угроз и управление трафиком.
- Фрагмент источника: глава 2, раздел «Proxy/Gateway».
- Область применения: архитектор, специалист по безопасности.

### SMAS-025. Доверие к PaaS как осознанное решение

- Утверждение: платформы (Kubernetes, Cloud Foundry, AWS) дают абстракции
  контроля доступа, но их использование требует доверия к самой платформе, что
  нельзя принимать легкомысленно; ответственность за доступ к веб-API часто всё
  равно остаётся за сервисом или делегированным посредником (например, шлюзом).
- Фрагмент источника: глава 2, разделы «PaaS», «Kubernetes».
- Область применения: архитектор, специалист по безопасности.

### SMAS-026. Развивающиеся подходы: OPA и service mesh

- Утверждение: контроль доступа всё больше уходит в платформу и выражается
  политиками высокого уровня (Open Policy Agent) и через service mesh (например,
  Istio с Envoy и взаимным TLS), что отделяет безопасность от отдельных
  сервисов.
- Фрагмент источника: глава 2, разделы «Emerging Approaches», «Service Mesh».
- Область применения: архитектор, специалист по безопасности.

## Глава 3. A General Approach to Microservice API Security (DHARMA)

### SMAS-027. Дуальность «доверенный/недоверенный» как приём оптимизации

- Утверждение: проверка доверия в рантайме обычно эффективнее аутентификации
  недоверенного источника, поэтому различие доверенных и недоверенных запросов
  можно использовать для оптимизации общего решения безопасности.
- Фрагмент источника: глава 3, раздел «Common Patterns…».
- Область применения: архитектор, специалист по безопасности.

### SMAS-028. Базовые понятия DHARMA

- Утверждение: `DHARMA` вводит трастовый домен (группа привилегированно
  взаимодействующих сервисов), доменное отношение (причина группировки),
  механизм доверия (проверка источника внутри домена) и механизм доступа
  (аутентификация и авторизация внешних запросов).
- Фрагмент источника: глава 3, раздел «DHARMA».
- Область применения: архитектор, специалист по безопасности.

### SMAS-029. Внутренние и граничные эндпойнты, доменная иерархия

- Утверждение: внутренний (interior) эндпойнт доступен сервисам внутри домена
  через механизм доверия, граничный (boundary) — внешним сервисам через механизм
  доступа; граничный эндпойнт одного домена может быть внутренним для внешнего
  домена, образуя доменную иерархию.
- Фрагмент источника: глава 3, раздел «DHARMA».
- Область применения: архитектор, специалист по безопасности.

### SMAS-030. Методология проектирования DHARMA из четырёх шагов

- Утверждение: методология DHARMA: (1) выявить трастовые домены, (2) определить
  механизмы доверия и доступа для каждого домена, (3) определить внутренние и
  граничные эндпойнты, (4) выбрать платформы реализации эндпойнтов.
- Фрагмент источника: глава 3, раздел «DHARMA Design Methodology».
- Область применения: архитектор, специалист по безопасности.

### SMAS-031. Платформонезависимая реализация в три уровня

- Утверждение: предложенная реализация — три уровня: внешняя область (внешние
  сущности), внешний домен (грубозернистые переиспользуемые сервисы) и
  внутренние домены (самые гранулярные сервисы), с разными механизмами для
  каждого уровня.
- Фрагмент источника: глава 3, разделы «Domain Hierarchy», «Trust and Access
  Mechanisms».
- Область применения: архитектор, специалист по безопасности.

### SMAS-032. Выбор механизмов по уровням

- Утверждение: для внешнего домена доступ — OAuth 2.0 с непрозрачным токеном;
  для внешнего домена доверие (и доступ внутренних доменов) — доверие на основе
  сертификатов и подписанных JWT; для внутренних доменов доверие смещено к
  производительности (изоляция сети, коллокация), но JWT может передаваться для
  учёта и наблюдаемости.
- Фрагмент источника: глава 3, раздел «Trust and Access Mechanisms».
- Область применения: архитектор, специалист по безопасности.

### SMAS-033. Соображения реализации: управление сертификатами и токенами

- Утверждение: реализация требует удостоверяющего центра для выдачи сертификатов
  (как минимум по одному на внутренний домен) и комплексного управления токенами
  (валидация, выпуск, обмен, разыменование), желательно с распределёнными
  защищёнными токен-сервисами для снижения задержки.
- Фрагмент источника: глава 3, разделы «Certificate management», «Token
  management».
- Область применения: специалист по безопасности, архитектор.

### SMAS-034. Короткоживущие токены и их область

- Утверждение: внутри доменов ожидаются короткоживущие JWT (срок жизни менее
  часа), возможно одноразовые; области OAuth (scopes) и утверждения JWT (claims)
  несут информацию для решений об авторизации.
- Фрагмент источника: глава 3, раздел «Token management».
- Область применения: специалист по безопасности, разработчик.

### SMAS-035. Аудит операций с токенами и решений об авторизации

- Утверждение: в распределённой архитектуре один запрос пользователя порождает
  несколько записей аудита; все операции с токенами (выпуск, обмен,
  распространение) и все решения об авторизации должны логироваться, обычно
  посредниками (шлюз, сервисный прокси).
- Фрагмент источника: глава 3, раздел «Microservice API accountability».
- Область применения: специалист по безопасности, архитектор.

### SMAS-036. Изоляция логики политик в посредниках

- Утверждение: чтобы изменение политики доступа не затрагивало всех
  разработчиков, логику применения политик следует изолировать в посредниках под
  управлением централизованных команд, а не раздавать в общих библиотеках;
  политики доступа стоит публиковать через OpenAPI.
- Фрагмент источника: глава 3, разделы «Publishing… Policies», «Access Control
  Policy Change Management».
- Область применения: архитектор, специалист по безопасности.

## Глава 4. Conclusion

### SMAS-037. Граница применимости и расширение DHARMA

- Утверждение: отчёт ограничен HTTP-API, но авторы считают, что абстрактная
  модель DHARMA может быть обобщена на контроль плоскости данных для других
  протоколов (gRPC, асинхронные сообщения), где зрелых механизмов доступа пока
  меньше.
- Фрагмент источника: глава 4, разделы «Applying DHARMA», «Extending DHARMA».
- Область применения: архитектор, специалист по безопасности.
