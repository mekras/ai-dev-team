## Структура отчёта

Отчёт решает одну задачу: предложить платформонезависимую модель управления
доступом к веб-API в микросервисной архитектуре. Он явно ограничен HTTP-API и не
рассматривает безопасность не-HTTP-протоколов и контейнеров в целом. Изложение
идёт от контекста (что такое микросервисы и какие у них качества) через обзор
существующих механизмов к собственной модели `DHARMA` и взгляду в будущее.

### Глава 1. Microservice Architecture

- Характеристики микросервиса: сервис-ориентированность, независимая
  развёртываемость и управляемость, эфемерность и эластичность; типичные
  технологии — веб-API (HTTP/JSON) и контейнеры.
- Словарь: сервис, экземпляр сервиса, интерфейс/веб-API, эндпойнт, запрос и
  ответ API, потребитель и поставщик API, посредник API (шлюз, сервисный
  прокси).
- Рамка управления доступом `IAAA`: идентификация, аутентификация, авторизация,
  аудит (accountability). В распределённой системе один запрос проходит через
  несколько решений об авторизации.
- Нефункциональные качества решения доступа: управляемость/операбельность,
  производительность (минимум добавленной задержки), удобство для разработчика.

### Глава 2. Access Control for Microservices

- Тезис «вся безопасность основана на доверии»: хорошая архитектура прозрачна
  относительно того, куда вложено доверие; безопасность — настройка баланса
  доверия и принятия риска.
- Сетевые механизмы: изоляция localhost, сегментация сети (зоны, ACL), SSL/TLS
  как обязательная мера для всех API; SPIFFE/SVID для идентичности сервисов.
- Прикладные механизмы (обмен токенами): ограничения классических веб-сессий и
  SAML (PDP/PEP, федерация); современные API-токены; API-ключи (только
  идентификация приложения, не авторитетны); OAuth 2.0 (делегирование
  авторизации, не аутентификация); OpenID Connect (слой аутентификации поверх
  OAuth, ID-токен); непрозрачные (by-reference) и прозрачные (by-value) токены;
  JWT (JWS/JWE) и ABAC на основе утверждений.
- Инфраструктура: прокси и API-шлюзы, обнаружение сервисов в рантайме, сетевые
  оверлеи; механизмы доступа в Kubernetes, Cloud Foundry (UAA), AWS (IAM).
- Развивающиеся подходы: Open Policy Agent (политики), service mesh (Istio,
  Envoy, mTLS), serverless (AWS Lambda).

### Глава 3. A General Approach to Microservice API Security (DHARMA)

- Общий приём: использовать дуальность «доверенный/недоверенный запрос» для
  оптимизации; проверка доверия в рантайме дешевле аутентификации недоверенного
  источника.
- Базовые понятия `DHARMA`: трастовый домен, доменное отношение, механизм
  доверия (внутри домена), механизм доступа (для внешних запросов), внутренние
  (interior) и граничные (boundary) эндпойнты, доменная иерархия.
- Методология проектирования из четырёх шагов: выявить трастовые домены →
  определить механизмы доверия и доступа → определить внутренние и граничные
  эндпойнты → выбрать платформы реализации.
- Платформонезависимая реализация в три уровня: внутренние домены (изоляция
  сети/коллокация, короткоживущие JWT), внешний домен (сертификаты, подписанные
  JWT), внешняя область (OAuth 2.0 с непрозрачным токеном).
- Соображения реализации: управление сертификатами и токенами, безопасное
  снабжение компонентов, развёртывание эндпойнтов через посредников, аудит всех
  операций с токенами и решений об авторизации.
- Опыт разработчика в DHARMA: что должен знать разработчик нового сервиса;
  публикация политик доступа через OpenAPI; изоляция логики политик в
  посредниках под управлением централизованных команд (а не в общих
  библиотеках).

### Глава 4. Conclusion: The Microservice API Security Frontier

- Предложение стандартизировать словарь микросервисной безопасности.
- Применение и расширение DHARMA, в том числе за пределы веб-API (gRPC,
  асинхронные протоколы) как обобщённый подход к контролю плоскости данных.
