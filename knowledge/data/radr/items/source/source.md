# Источник: `RADR`

## Статус

Источник принят в корпус знаний как локально сохранённая рабочая копия книги о
правилах проектирования REST API с ограничением на публикацию полного текста в
Git.

## Происхождение

- Название: `REST API Design Rulebook`
- Автор: Mark Massé
- Издатель: O’Reilly Media, Inc.
- Год издания: `2012` (первый выпуск — `2011-10-17`)
- ISBN: `978-1-449-31050-9`
- Тип источника: локальная PDF-книга (англоязычная)
- Способ получения: по решению пользователя в проект скопирована локальная
  рабочая копия книги с хранением файла только в игнорируемом каталоге
  `knowledge/.local/raw/radr/`
- Дата включения в корпус: `2026-06-24`

## Что сохранено

Для локальной обработки получены следующие артефакты:

- паспорт: `knowledge/data/radr/items/source/source.md`
- индекс артефактов: `knowledge/data/radr/items/source/page-index.tsv`

В Git сохраняются только метаданные источника, обзорный нормализованный слой и
извлечённые утверждения без крупных дословных фрагментов книги.

## Условия использования

- Открытая лицензия на повторное использование полного текста не установлена.
- Права принадлежат автору и издателю O’Reilly Media, Inc.
- Уровень уверенности: средний.
- Полный PDF и полная текстовая выгрузка не публикуются в Git публичного
  проекта.
- Допустимое использование в корпусе: локальный анализ, нормализация тем,
  извлечение собственных кратких утверждений и ссылок на артефакты.
- Ограничение: не публиковать полный текст и крупные дословные фрагменты без
  отдельного правового основания.
- Атрибуция и правовой статус: `knowledge/source-attribution.md`.

## Ограничения

- Это локальная рабочая копия книги, а не ссылка на издательский сайт.
- Текстовая выгрузка получена из PDF утилитой `pdftotext -layout`; OCR не
  применялся, прозаический текст читается без искажений символов.
- Примеры кода, таблицы кодов состояний и врезки в текстовой выгрузке
  представлены частично или с искажением порядка; при ссылке на них нужно
  сверяться с PDF.
- Книга датирована `2012` годом; отдельные технические рекомендации (например,
  по кросс-доменному доступу и версионированию) следует сверять с современной
  практикой.
- Извлечённые утверждения нельзя считать решениями проекта без отдельного
  анализа влияния.

## Назначение в проекте

Источник может использоваться как первичный материал для:

- проектирования идентификаторов ресурсов (URI): разделители, архетипы ресурсов
  (document, collection, store, controller), фильтрация и пагинация через query;
- проектирования взаимодействия по HTTP: назначение методов и корректный выбор
  кодов состояния 2xx/3xx/4xx/5xx;
- проектирования метаданных: заголовки представления и кэширования, медиатипы и
  их согласование;
- проектирования представлений: JSON, гипермедиа (HATEOAS), единообразные формы
  ссылок, связей и ошибок;
- учёта потребностей клиентов: версионирование, безопасность (OAuth, API
  management), частичные ответы, кросс-доменный доступ (JSONP, CORS);
- развития навыка `software-architecture` и смежных навыков в части
  проектирования веб-API.
