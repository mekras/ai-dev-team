## Назначение

Документ фиксирует утверждения по `OWASP ASVS 4.0.3`, достаточные для работы
роли `security-engineer` с требованиями и верификацией безопасности приложений.
Утверждения фиксируют содержание стандарта, а не решения репозитория. Опора —
локальная текстовая выгрузка PDF. Все утверждения имеют статус
`ready_for_review`. Лицензия источника — `CC BY-SA 4.0` (требуется атрибуция).

## Назначение и уровни

### ASVS-001. Назначение стандарта

- Утверждение: ASVS задаёт открытый каталог проверяемых требований безопасности
  веб-приложений и сервисов и служит основой для их верификации; его можно
  использовать как метрику, руководство и основу для контрактных требований.
- Фрагмент источника: введение.
- Область применения: специалист по безопасности, аналитик, архитектор.

### ASVS-002. Три уровня верификации

- Утверждение: стандарт задаёт три уровня доверия: L1 (минимальный, полностью
  проверяем «снаружи», в т. ч. пентестом), L2 («Standard», для приложений с
  чувствительными данными, типичная цель), L3 («Advanced», для самых критичных
  приложений).
- Фрагмент источника: раздел «Application Security Verification Levels».
- Область применения: специалист по безопасности, project-manager.

### ASVS-003. L1 как минимум для всех

- Утверждение: L1 — минимум, к которому должны стремиться все приложения, и
  полезен как первый шаг; для приложений с ценными данными редко стоит
  останавливаться на L1.
- Фрагмент источника: описание Level 1.
- Область применения: специалист по безопасности.

### ASVS-004. Глубина проверки растёт с уровнем

- Утверждение: L2 и L3 требуют доступа к коду, документации и разработчикам и
  более глубокого анализа архитектуры, кода и тестирования, чем L1.
- Фрагмент источника: описания Level 2 и Level 3.
- Область применения: специалист по безопасности, роль проверки результата.

## Разделы требований

### ASVS-005. Четырнадцать разделов требований

- Утверждение: требования сгруппированы в 14 разделов (V1–V14): V1 Architecture,
  Design and Threat Modeling; V2 Authentication; V3 Session Management; V4
  Access Control; V5 Validation, Sanitization and Encoding; V6 Stored
  Cryptography; V7 Error Handling and Logging; V8 Data Protection; V9
  Communication; V10 Malicious Code; V11 Business Logic; V12 Files and
  Resources; V13 API and Web Service; V14 Configuration.
- Фрагмент источника: оглавление разделов.
- Область применения: специалист по безопасности, разработчик.

### ASVS-006. Моделирование угроз как раздел требований

- Утверждение: раздел V1 объединяет архитектуру, проектирование и моделирование
  угроз, связывая требования безопасности с архитектурными решениями.
- Фрагмент источника: раздел V1.
- Область применения: специалист по безопасности, архитектор.

### ASVS-007. Сопоставление требований с уровнями и CWE

- Утверждение: каждое требование помечено применимыми уровнями (L1/L2/L3) и
  сопоставлено с идентификатором слабости CWE, что связывает стандарт с
  таксономией уязвимостей и делает требования проверяемыми.
- Фрагмент источника: формат таблиц требований.
- Область применения: специалист по безопасности, разработчик, роль проверки.

### ASVS-008. Связь с требованиями и приёмкой

- Утверждение: ASVS можно использовать для вывода проверяемых требований
  безопасности и критериев приёмки приложения; L1 задуман как набор «essential»
  контролей.
- Фрагмент источника: введение и описание уровней.
- Область применения: аналитик, специалист по безопасности.
