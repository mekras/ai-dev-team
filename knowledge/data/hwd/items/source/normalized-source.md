# Нормализованное представление: `HWD`

## Основание

- Источник учёта: `knowledge/data/hwd/source.yml`
- Паспорт источника: `knowledge/data/hwd/items/source/source.md`
- Индекс первичных артефактов: `knowledge/data/hwd/items/source/page-index.tsv`
- Снимок материала: `knowledge/data/hwd/items/`
- Commit снимка: `287c3b86170f56b163dc65eeaaf3678c3cefb25c`

## Назначение нормализации

Этот слой подготавливает компактное и последовательное представление структуры
источника `HWD` для последующего извлечения утверждений.

Нормализация не заменяет исходный снимок и не пересказывает содержание всех
файлов. Она приводит структуру источника к форме, удобной для обзора и
дальнейшей аналитической работы.

## Тип материала

- Основной формат содержательных файлов: `rst`
- Дополнительные форматы: `md`, `png`, файл `README` без расширения
- Общий характер источника: документационный корпус со вложенной структурой
  разделов

## Степень зрелости источника

- в `index.rst` источник прямо помечен как находящийся в процессе написания;
- состав, расположение и наполнение статей могут существенно меняться;
- в корпусе есть страницы с явными `todo`, то есть часть замысла источника ещё
  не раскрыта содержательно;
- поэтому источник пригоден для извлечения принципов, модели организации
  документации и отдельных проектных ходов, но не должен использоваться как
  полностью завершённый свод правил.

## Верхний уровень структуры

- `_include`
- `_static`
- `appendix`
- `guides`
- `references`
- `theory`
- `index.rst`
- `intro.rst`
- `legal.rst`
- `resources.rst`

## Наблюдаемая структура разделов

### `appendix`

- есть отдельный индекс раздела;
- есть ветка `quick-solution`;
- есть ветка `templates`;
- внутри встречаются более глубокие продуктовые и прикладные разветвления.

### `guides`

- есть отдельный индекс раздела;
- есть набор самостоятельных guide-страниц;
- есть вложенные подкаталоги `guide`, `intro`, `product`, `readers`.

### `references`

- это самый разветвлённый раздел источника;
- внутри выделены тематические ветки `articles`, `formats`, `kb`, `locations`,
  `packages`, `readers`, `sections`, `structure`, `tools`, `types`;
- у большинства веток есть собственные `index.rst`.

### `theory`

- есть отдельный индекс раздела;
- есть набор теоретических страниц: `goal`, `how-people-read`, `principles`,
  `why`.

## Наблюдаемая смысловая схема

- `intro.rst` задаёт назначение корпуса: помощь разработчикам, техническим
  писателям и другим участникам разработки, которым приходится писать
  документацию;
- основной контур корпуса делится на три главных раздела:
  - `theory` для целей, принципов и общего понимания документации;
  - `guides` для пошаговых руководств и прикладных приёмов;
  - `references` для кратких справочных материалов без развёрнутых пояснений;
- дополнительно присутствуют входные точки для начала работы:
  - `appendix/quick-solution` для быстрого поиска решения;
  - `guides/startup.rst` для первого входа в задачу написания документации;
- в `intro.rst` явно описана парность многих материалов: руководство раскрывает
  тему подробно, а справочник даёт краткую выжимку и обратные ссылки на
  применение.

## Наблюдаемые структурные признаки

- источник организован как иерархия документационных разделов;
- в источнике выделены самостоятельные верхнеуровневые тематические области
  `appendix`, `guides`, `references`, `theory`;
- индексные страницы используются системно;
- темы разделены по типам материалов и по назначению;
- часть структуры ориентирована на читателей, часть на форматы, часть на виды
  материалов;
- в источнике есть как содержательные страницы, так и вспомогательные шаблоны и
  статические материалы.

## Наблюдаемые содержательные опоры

Ниже — наш производный обзор содержательных тем источника по его структуре,
изложенный своими словами по смыслу первоисточника.

### Назначение и аудитория корпуса (`intro.rst`)

- корпус адресован разработчикам, техническим писателям и другим участникам
  разработки, которым приходится писать документацию;
- в источнике документация рассматривается как часть процесса разработки ПО
  наряду с требованиями, заданиями, правилами, справочниками и руководствами;
- хорошая документация рассматривается как средство ускорения разработки и
  повышения привлекательности продукта, а плохая или отсутствующая документация
  — как источник ошибок и отказа от использования продукта;
- основная часть корпуса сознательно разделена на три разных типа материалов:
  `theory`, `guides`, `references`;
- в корпусе описана парная связь между руководствами и справочниками:
  руководство раскрывает тему подробно, а справочник даёт краткую выжимку и
  ссылки на применение.

### Назначение документации (`theory/goal.rst`)

- `theory/goal.rst` задаёт центральную идею: назначение документации в этом
  источнике сформулировано как помощь читателям в достижении их целей;
- источник различает цели и задачи и утверждает, что задачи существуют как
  препятствия на пути к целям, а не как самостоятельная ценность;
- документация должна по возможности избавлять читателя от задач, а если это
  невозможно, помогать решать их с наименьшими затратами.

### Принципы (`theory/principles.rst`)

- `theory/principles.rst` разворачивает из этой идеи набор производных
  принципов: забота о читателе, экономия его времени, находимость, ссылки,
  примеры, руководства для типичных задач, ориентация на одну аудиторию;
- источник считает принципы более фундаментальным уровнем, чем советы,
  руководства, правила и частные приёмы;
- цели читателя объявлены главным критерием для того, что, как и где писать в
  документации;
- в качестве производных практик из заботы о читателе источник отдельно выделяет
  экономию времени читателя, повышение находимости, использование ссылок,
  примеров и руководств для типичных задач;
- источник продвигает принцип одной аудитории, то есть рекомендует писать
  документ для одной конкретной читательской аудитории, а не для всех сразу.

### Как люди читают документацию (`theory/how-people-read.rst`)

- `theory/how-people-read.rst` описывает модель поведения читателя: документацию
  читают ради решения задачи, ищут по поиску или структуре и часто читают
  выборочно;
- источник исходит из того, что люди читают документацию для решения задачи и
  обычно прекращают чтение после того, как задача решена или кажется решённой;
- в источнике выделены три способа поиска нужной информации в документации:
  автоматизированный поиск, ручной поиск по структуре и последовательное чтение
  всей документации.

### Начало работы (`guides/startup.rst`)

- `guides/startup.rst` задаёт начальный рабочий порядок для новой документации и
  новой статьи;
- `guides/startup.rst` предлагает начальный рабочий порядок для новой
  документации: определить круг читателей, место размещения и формат;
- `guides/startup.rst` предлагает отдельный порядок начала новой статьи:
  определить требования, составить план, написать текст и вычитать статью;
- при определении требований к новой статье источник требует ответить как
  минимум на три вопроса: для кого статья, о чём она и какой вид статьи нужен.

### Читательская аудитория (`guides/readers.rst`)

- `guides/readers.rst` предлагает сначала разделить аудиторию документации на
  внутреннюю и внешнюю, а затем уточнять подтипы внутри каждой ветви;
- для внутренних читателей источник отдельно перечисляет как минимум команду
  продукта, команды других продуктов, внутренних пользователей и прочих
  внутренних заинтересованных лиц;
- для внешних читателей источник отдельно различает конечных пользователей и
  сторонних разработчиков;
- `guides/readers.rst` задаёт полезную классификацию аудиторий, но значительная
  часть детальных подразделов ещё не заполнена содержанием.

### Зрелость и формат

- часть guide-страниц существует пока как заготовки темы, а не как завершённые
  руководства;
- часть заявленных тем в `guides` пока существует только как заготовка с `todo`,
  поэтому по этим темам источник нельзя использовать как содержательно
  завершённое руководство;
- для корпуса характерна системная опора на `rst` как основной формат
  документации.

## Ограничения нормализованного слоя

- этот слой описывает структуру, а не полный смысл всех документов;
- он не определяет, какие идеи источника следует переносить в каркас;
- он не заменяет исходные файлы при последующей проверке утверждений;
- статические и вспомогательные материалы включены только как элементы
  структуры, без интерпретации их роли по содержанию;
- наличие `todo` в части guide-страниц означает, что для некоторых тем источник
  задаёт направление, но ещё не даёт содержательного основания для производных
  правил.
