## Почему источник относится к проекту

Источник рассматривается как методическое основание для:

- структуры и содержания технического задания на программное обеспечение;
- правил формулирования требований к программе и программному изделию по ГОСТ
  19.201-78 и смежным стандартам;
- описания разделов проектной документации: введение, основания, назначение,
  технические требования, стадии и этапы, порядок контроля и приёмки.

## Условия использования

- Открытая лицензия на повторное использование полного текста не установлена.
- Авторы содержания указаны на титульной странице; поле «автор» в метаданных PDF
  отражает изготовителя файла, а не правообладателя содержания.
- Уровень уверенности: средний.
- Хранение полного PDF и полной текстовой выгрузки в Git публичного проекта: не
  допускается.
- Локальное хранение рабочего PDF и его текстовой выгрузки: допустимо в рамках
  выбранного проектом режима с исключением из Git.
- Допустимое использование: локальный анализ, обзор тем, извлечение кратких
  проверяемых утверждений и производных выводов без крупных дословных
  фрагментов.
- Атрибуция и правовой статус: `knowledge/source-attribution.md`.

## Что уже известно по источнику

- Пособие описывает типовую структуру и содержание разделов технического задания
  на программное обеспечение в соответствии с ГОСТ 19.201-78.
- Документ разбирает разделы введения, оснований для разработки, назначения
  разработки, технических требований к программе, технико-экономических
  показателей, стадий и этапов разработки, порядка контроля и приёмки и
  приложений.
- Документ выделяет типы ПО: прикладное, системное, ПО Web-систем, ПО
  специализированных компьютерных систем и информационные системы с СУБД.
- Текстовая выгрузка из PDF пригодна для первичного извлечения утверждений, но
  требует сверки с PDF в спорных местах.

## Что можно делать дальше

- расширять набор утверждений по отдельным разделам ТЗ при необходимости;
- сопоставить с источниками `SREQ` и `SMFR` и навыками проекта по требованиям;
- при необходимости подготовить отчёт влияния на навыки требований.

## Что требует осторожности

- не публиковать в Git полный PDF, полную текстовую выгрузку и крупные дословные
  фрагменты;
- не выдавать тезисы пособия за принятые решения проекта;
- не считать поле «автор» в метаданных PDF подтверждением прав на содержание;
- различать наблюдения из пособия, интерпретации агента и локальные правила
  этого репозитория.
