## Структура книги

Книга построена как свод из примерно `80` именованных правил (`Rule:`) для
проектирования веб-ориентированных REST API; каждое правило сохраняет
модальность (must / should / may) и сопровождается обоснованием и примерами.
Степень «RESTful» оценивается по модели зрелости Леонарда Ричардсона с тремя
уровнями: URI, HTTP и Hypermedia. Автор вводит собственную концептуальную рамку
WRML (Web Resource Modeling Language) с медиатипом `application/wrml`, у
которого подключаемые компоненты format и schema, и применяет её, чтобы задать
единообразные формы для ссылок, связей, схем и ошибок.

### Глава 1. Introduction

История и архитектура Веба, постановка задачи единообразного дизайна. Темы:
шесть ограничений архитектурного стиля Веба (client-server, uniform interface,
layered system, cache, stateless, code-on-demand); четыре ограничения
единообразного интерфейса (идентификация ресурсов, манипуляция через
представления, самоописываемые сообщения, HATEOAS); REST как описание стиля по
Филдингу; модель зрелости Ричардсона; концепция WRML.

### Глава 2. Identifier Design with URIs

Правила построения идентификаторов ресурсов: разделитель `/` для иерархии, отказ
от завершающего `/`, дефисы вместо подчёркиваний, нижний регистр, отказ от
расширений файлов; согласованные субдомены (`api.`, `developer.`); архетипы
ресурсов и их имена (существительное в единственном числе для document, во
множественном — для collection и store, глагол для controller); запрет имён
CRUD-функций в URI; переменные сегменты URI; query для фильтрации и пагинации
(`pageSize`, `pageStartIndex`).

### Глава 3. Interaction Design with HTTP

Назначение HTTP-методов и кодов состояния: запрет туннелирования через GET/POST;
GET (получение), HEAD (заголовки), PUT (вставка и обновление в store, обновление
mutable-ресурсов), POST (создание в collection и запуск controller), DELETE
(удаление), OPTIONS (метаданные взаимодействий); коды 2xx (200, 201, 202, 204),
3xx (301, 302 — не использовать, 303, 304, 307), 4xx (400, 401, 403, 404, 405,
406, 409, 412, 415), 5xx (500).

### Глава 4. Metadata Design

Метаданные через HTTP-заголовки и медиатипы: `Content-Type`, `Content-Length`,
`Last-Modified`, `ETag`, `Location`; условные запросы к store
(`If-Unmodified- Since`, `If-Match`); кэширование (`Cache-Control`, `Expires`,
`Date`, `Pragma`), поощрение и подавление кэша, негативное кэширование; запрет
менять поведение методов через кастомные заголовки; синтаксис и виды медиатипов
(зарегистрированные, vendor-specific `vnd`, `application/wrml`); согласование
медиатипа через `Accept` и через query-параметр `accept`; версионирование схем.

### Глава 5. Representation Design

Структура тела ответа и гипермедиа: JSON как формат представления и требование
well-formed JSON; опциональные XML/HTML; запрет дополнительных «конвертов»;
единообразная форма ссылок (link), связей (link relation) и их публикации;
обязательная ссылка `self`; минимизация числа «entry point» URI и опора на
docroot; ссылки как контекстно-зависимое меню действий (HATEOAS); единообразная
форма ошибки (`id`, `description`), контейнера ошибок (`ErrorContainer`) и общих
типов ошибок.

### Глава 6. Client Concerns

Потребности клиентов: версионирование (новый URI для нового концепта, версии
формы через схемы, версии состояния через ETag; отказ от `v2` в URI);
безопасность (OAuth, API management / reverse proxy); частичные ответы и
композиция (query-параметр `fields`, встраивание связанных ресурсов через
query); кросс-доменный доступ из JavaScript под same origin policy (JSONP —
только чтение через GET; CORS — чтение и запись для всех методов с preflight).
