## Границы аудита

Проверялось влияние расширенного официального источника `CODX` на:

- знания проекта о `AGENTS.md` и configuration layers Codex;
- guidance по sandbox, approvals и permissions;
- guidance по MCP, skills, subagents и automation;
- производные документы текущего проекта.

Аудит не пересматривал весь проект по всем возможным страницам Codex. В фокус
вошли только страницы, содержащие сведения о настройке, использовании,
автоматизации и особенностях Codex.

## Существенные сведения источников

- `CODX-001`..`CODX-004` — `AGENTS.md` и project instructions имеют явную
  модель обнаружения, объединения и лимита по размеру.
- `CODX-005`..`CODX-009` — конфигурация Codex многослойна; project-scoped
  `.codex/` загружается только в trusted projects; sandbox/approval и
  permission profiles имеют отдельные правила и ограничения.
- `CODX-010`..`CODX-011` — MCP является first-class механизмом подключения
  внешних инструментов и контекста, с отдельными настройками transport,
  approvals и server instructions.
- `CODX-012`..`CODX-014` — skills в Codex используют progressive disclosure,
  имеют жёсткие ограничения на начальный metadata-budget и сканируются из
  `.agents/skills` по пути к корню репозитория.
- `CODX-015`..`CODX-016` — subagents запускаются только по явному запросу и
  имеют отдельные ограничения по fan-out и глубине.
- `CODX-017`..`CODX-019` — `codex exec`, SDK и app-server закрывают разные
  сценарии automation и integration и не должны смешиваться как одна и та же
  поверхность.
- `CODX-020` — обязательные project rules должны жить в `AGENTS.md` или
  checked-in docs; memories — только вспомогательный слой локального recall.

## Расхождения и решения

### Severity: Medium — `knowledge/statements/CODX.md` был существенно уже реального источника

Документ: `knowledge/statements/CODX.md`

Источник: `CODX`

Класс: устаревшее утверждение / неполное покрытие

Факт источника: ранее в `CODX` был отражён только `guides/agents-md`, тогда как
официальный раздел Codex содержит значимый материал по конфигу, sandbox,
permissions, MCP, skills, subagents, non-interactive mode, SDK и app-server.

Вывод: слой утверждений нужно расширить с одной темы до набора проверяемых
тем, которые реально влияют на навыки и правила проекта.

Что сделать: обновить `knowledge/statements/CODX.md`.

### Severity: Low — карточка и паспорта источника описывали `CODX` как одну страницу

Документ: `knowledge/inventory/CODX.md`, `knowledge/primary/CODX/source.md`,
`knowledge/normalized/CODX/source.md`, `knowledge/source-attribution.md`

Источник: `CODX`

Класс: устаревшее описание источника

Факт источника: источник уже технически расширен до набора страниц.

Вывод: документы учёта должны описывать `CODX` как набор официальных страниц
Codex documentation, а не как единичную статью.

Что сделать: обновить source-passport и inventory.

### Severity: Low — прямого конфликта с текущими проектными правилами не выявлено

Документ: `AGENTS.md`, `README.md`, навыки проекта

Источник: `CODX`

Класс: изменение без обязательной правки

Факт источника: документация Codex подтверждает и уточняет общую модель
`AGENTS.md`, skills, MCP и sandbox, но не требует немедленно ослаблять или
переписывать текущие project-specific правила этого репозитория.

Вывод: производные правила проекта можно оставить без изменений, если они
остаются осознанно более строгими и привязанными к локальному процессу.

Что сделать: зафиксировать влияние в knowledge-слое и при необходимости
использовать новые утверждения при следующих правках навыков.

## Предлагаемые изменения

- Обновить `knowledge/statements/CODX.md`.
- Зафиксировать отчёт влияния для расширенного `CODX`.
- Не менять `README.md`, `AGENTS.md` и навыки автоматически только по факту
  появления новых официальных страниц, если нет прямого проектного противоречия.

## Ограничения проверки

- Полные HTML-снимки страниц не публикуются в Git и используются только как
  локальная первичная опора.
- Часть страниц раздела Codex носит справочный или быстро меняющийся характер
  (`changelog`, surface-specific pages, enterprise/security variants); текущий
  отчёт фиксирует только те сведения, которые выглядят устойчивыми и
  потенциально значимыми для проектных правил.
