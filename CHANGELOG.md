# История изменений

Все заметные изменения этого проекта документируются в этом файле.

Формат основан на [Keep a Changelog](https://keepachangelog.com/ru/1.1.0/),
проект использует [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

Раздел «Действия при обновлении» перечисляет шаги, которые потребителю продукта
нужно выполнить в своём проекте при переходе на эту версию. Раздел появляется
только когда выпуск действительно требует таких действий.

## [Невыпущено]

### Добавлено

- Навык `ait-req-revalidation`: сквозная постепенная перепроверка требований по
  запросам вида «Перепроверь требования». Ведёт проход по слоям от первичных
  данных корпуса знаний и концепции через высокоуровневые требования к
  детальным, закрывает каждый слой до перехода к следующему и запускает анализ
  влияния на внесённые изменения, делегируя предметную проверку профильным
  навыкам.

## [0.11.1] - 2026-06-29

### Изменено

- Обновлены зависимости APM: `ai-russian-language` до `^1.2.6`,
  `project-knowlege-corpus` до `^0.7.0`.
- Корпус знаний: проверена полнота извлечения источника SREQ, извлечены
  приложения, утверждения сверены с источником и переведены в статус `verified`.

## [0.11.0] - 2026-06-29

### Добавлено

- В корпус знаний добавлен источник `ADRG` (`adr.github.io`).
- Слой согласования понятий `knowledge/concepts.yml` между корпусом и навыками
  (решение — `docs/adr/0001-concepts-reconciliation-layer.md`, реестр ADR
  проекта `docs/adr/`).
- Навык `ait-architecture`: справки «Архитектурные силы и компромиссы»
  (`architectural-forces.md`), «Подходы к декомпозиции» (`decomposition.md`) и
  каталог шаблонов ADR (`adr-templates.md`).
- Навыки требований: чек-лист характеристик качества ГОСТ 25010
  (`quality-characteristics.md`) и требования сопровождаемости
  (`maintainability-requirements.md`).

### Изменено

- Навык `ait-architecture` существенно дополнен по источнику `SAHP`: анализ сил
  и компромиссов как центральная компетенция; правила создания и ведения ADR,
  выбор шаблона под случай и статусы ADR; декомпозиция (выбор подхода и шесть
  паттернов); операционные данные (силы разделения, пять этапов, выбор типа БД,
  распределённый доступ к данным); гранулярность сервисов с критериями;
  повторное использование кода; транзакционные саги; характеристики качества
  (сопровождаемость, развёртываемость, масштабируемость, адаптируемость к
  пиковым нагрузкам); термин «сопряжение» вместо «связанности», афферентное и
  эфферентное сопряжение.
- Навыки `ait-req-*` проверяют полноту требований по модели качества ГОСТ 25010
  и записывают требования сопровождаемости.
- Навык `ait-code-testing` ведёт реализацию функций пригодности как исполняемых
  проверок в CI/CD, замыкая цепочку «решение → функция пригодности → проверка».

## [0.10.2] - 2026-06-28

### Изменено

- `ait-changelog` и правила выпуска теперь требуют кратко описывать небольшие
  изменения документации без пересказа редакторских подробностей.
- Добавлена роль `documentation-reader` для проверки пользовательской
  документации глазами читателя, а `ait-writing` и маршрутизация ролей теперь
  используют её перед приёмкой такой документации.

### Исправлено

- `ait-writing` теперь ведёт серию пользовательской документации на верхнем
  уровне через технического писателя, создаёт файлы по согласованному режиму
  приёмки и использует соседние навыки только для конкретных подзадач.
- `ait-setup` больше не выносит `apm compile --dry-run` как обычный следующий
  шаг настройки проекта с рукописными клиентскими инструкциями.
- `ait-setup` больше не объявляет настройку завершённой после частичных
  стартовых правок без статуса по обязательным направлениям настройки.
- `ait-writing` больше не должен записывать планы работ в проект без явного
  запроса, терять замечания пользователя между файлами и дублировать соседние
  документы без пользы для читателя.

## [0.10.0] - 2026-06-27

### Исправлено

- Ветка обновления версии в навыке `ait-setup` больше не подменяет пересмотр
  проекта повторным обновлением зависимостей APM.
- Ветка обновления версии больше не считает историю `apm.lock.yaml`
  доказательством, что проект уже был пересмотрен под предыдущую версию
  продукта.
- Добавлены проверки срабатывания и сценарии результата для команды «Обнови
  проект под новую версию ai-dev-team.».

### Добавлено

- Навык `ait-changelog` для создания, проверки и сопровождения журнала изменений
  целевого проекта.
- Навык `ait-setup` теперь проверяет наличие журнала изменений в проекте,
  предлагает создать `CHANGELOG.md` при его отсутствии и закрепляет решение о
  журнале в `AGENTS.md`.
- Журнал изменений `CHANGELOG.md` с подразделом «Действия при обновлении» как
  путь пересмотра проекта-потребителя при переходе на новую версию продукта.
- Ветка обновления версии в навыке `ait-setup`: определение перехода версий по
  `apm.lock.yaml`, чтение журнала установленной копии продукта и проведение
  человека по действиям с подтверждением каждого шага.
- Раздел «Обновление продукта» в `README.md` и дисциплина выпуска в `AGENTS.md`
  и `docs/development.md`.

### Действия при обновлении

- Для примитива `ait-setup`: после установки новой версии выполните команду
  «Обнови проект под новую версию ai-dev-team.». Навык создаст или обновит
  `.ai-dev-team-state.yml`, где фиксируется версия `ai-dev-team`, под которую
  проект уже пересмотрен.
- Для примитивов `ait-setup`, `AGENTS.md`: при следующей настройке проекта
  решите, нужен ли журнал изменений проекта. Если он нужен, подтвердите создание
  `CHANGELOG.md` и короткое правило его сопровождения через `ait-changelog`.
