# Правила для агентов

Этот файл задаёт локальные правила для агентов ИИ, которые помогают
разрабатывать **сам продукт**.

## Видимость проекта

Этот проект является публичным.

## Общие правила

- При написании русских текстов используй навык `ru-lang`.
- При работе с исходными кодами и VCS используй навык `ru-dev`.
- Формат файлов в проекте должен соблюдать требования файла `.editorconfig`.
- Для файлов Markdown соблюдай длину строки не более 80 символов. Перед записью
  файла применяй `prettier` для автоматического форматирования.
- Файлы отчётов создавай только если это явно требует навык или пользователь.

## Корпус знаний проекта

Проект использует корпус знаний в `knowledge/`.

Перед значимыми изменениями в коде, документации, требованиях, правилах,
архитектуре или пользовательских сценариях проверяй, есть ли в корпусе знаний
релевантные источники, утверждения, отчёты влияния или открытые вопросы.

Используй корпус знаний как проверяемое основание для разработки проекта:

- сначала опирайся на источники, утверждения и ограничения из корпуса;
- профильные навыки перед содержательной работой проверяют корпус знаний проекта
  и не заменяют его встроенными справочниками из коллекции навыков;
- не выдавай выводы из корпуса за принятое решение, если требуется решение
  владельца проекта;
- если корпус противоречит задаче пользователя или текущим файлам проекта, явно
  зафиксируй противоречие и не скрывай его;
- если работа меняет смысловые сведения проекта, проверь, нужно ли обновить
  корпус знаний или выполнить анализ влияния;
- если источник нельзя копировать, он слишком большой, часто меняется или
  доступен через права проекта, предложи изменить способ работы с ним:
  индексировать, получать точечно или обращаться к первоисточнику вместо
  хранения полной копии;
- если корпус отсутствует, неполон или структурно повреждён, сообщи об этом как
  об ограничении и не имитируй проверку корпуса.

Для проверки состояния корпуса используй навык `kc-validation`. Для движения
данных от источников к изменениям проекта используй `kc-pipeline` и соседние
навыки корпуса знаний.

## Проверки

- Для полной проверки коллекции APM используй `apm run tests`.

## Экономное использование подагентов

Цель маршрутизации подагентов: снизить расход основных лимитов и сохранить
качество за счёт узкого делегирования, проверки и редкой эскалации.

- Обычную работу, финальный ответ и приёмку результата выполняет родительский
  агент на `gpt-5.4`, если пользователь или среда не задали другую модель.
- Для узкого поиска, извлечения фактов, списков кандидатов и малых сводок
  сначала используй `spark_evidence` на `gpt-5.3-codex-spark`: эта модель имеет
  отдельный лимит и не расходует общий пул `gpt-5.4`, `gpt-5.4-mini` и
  `gpt-5.5`.
- Используй `mini_mechanical` на `gpt-5.4-mini` только для низкорисковых
  механических задач внутри общего лимита, когда `spark_evidence` недоступен или
  явно нужен проход малой модели из общего пула.
- Используй `strong_reviewer` на `gpt-5.5` только для редкой независимой
  проверки правил, ролей, навыков, архитектурных решений, источников истины и
  других изменений с последствиями для будущих задач.
- Не делегируй микрозадачи, широкие решения, финальную приёмку результата,
  работу с неясными критериями или изменения файлов без явного контракта.
- Максимум: два параллельных подагента и один уровень вложенности. Дочерний
  агент не должен запускать собственных подагентов без отдельного решения
  родительского агента.
- Каждый результат подагента должен быть структурирован и содержать `status`,
  `items`, `evidence`, `confidence`, `risk`, `open_questions` и
  `escalation_needed`.
- Родительский агент отклоняет результат подагента, если тот расширил область,
  не дал ссылок на файлы или источники, сообщил о неуверенности, столкнулся с
  противоречием или потребовал больше одной повторной попытки.
- Для гарантированного выбора модели используй проектные запускатели
  `tools/codex-model-subagent`, `tools/codex-spark-subagent` и проверяй расход
  через `tools/codex-usage-report`.

После каждой задачи используй правила `ai-setup-subagents` для однострочной
сводки моделей и лимитов.

## Внесение изменений

- Разрешается вносить изменения только внутри папки проекта.

## Git

- Все изменения, даже чужие, если они не исключены через `.gitignore`, сразу
  добавляй в индекс Git.
- Не делай commit и push без явной просьбы пользователя.

## Выпуск версии

Перед тегом новой версии обновляй `CHANGELOG.md`:

- перенеси накопленные записи из раздела «Невыпущено» в раздел новой версии с
  датой;
- для небольших изменений документации пиши кратко по новому ориентиру для
  читателя; не расписывай переносы, сокращения и перестановки текста, если они
  не дают отдельного последствия для потребителя;
- если выпуск требует действий от проектов-потребителей (переименование или
  удаление навыка, агента или правила, изменение структуры проекта или корпуса),
  добавь в раздел версии подраздел «Действия при обновлении» с привязкой к
  примитиву;
- если действий в проекте-потребителе не требуется, подраздел не добавляй.

Раздел «Действия при обновлении» — источник для ветки обновления версии в навыке
`ait-setup`. Не оставляй значимые для потребителя изменения без записи.

## Ограничения

Не смешивай переносимое ядро продукта из `.apm/` с локальными особенностями
этого репозитория.

Не переписывай смысл документов ради стилистического улучшения.

Не создавай крупную структуру файлов без предварительного обоснования через
проблему, риск и проверяемую пользу.

Если задача выглядит неподходящей для ИИ или недостаточно определённой, сначала
укажи риск и предложи безопасный следующий шаг.
