# Проектирование API

Используй этот файл, когда задача — спроектировать или развивать прикладной или
веб-API: интерфейс, которым пользуются другие команды или внешние разработчики.
Это стилезависимая справка, а не выбор по умолчанию: если у системы нет
публичного или межкомандного контракта, опирайся на `component-design.md` и
`quality-criteria.md`. REST, HTTP и JSON здесь — частая, но не единственная
форма; принципы проектирования и совместимости применимы и к другим протоколам.

API — это формальное обязательство связывать программируемые контексты. Его
потребители — инженеры, а ошибки и нарушения совместимости умножаются на число
потребителей. Поэтому цена ошибки в API выше, чем в обычном коде: плохое имя
исправить просто, неверно понятое назначение — почти невозможно.

## Проектирование сверху вниз

Проектируй API от смысла к деталям, а не наоборот. Порядок шагов и цена ошибки
растут к вершине:

1. **Область применения.** Сначала определи, какую проблему и для кого решает
   API. Проверь решение, задав вопрос «какую проблему мы решаем?» с ударением по
   очереди на каждом слове: нужна ли задача, наша ли это компетенция, адекватно
   ли решение.
2. **Уровни абстракции.** Раздели уровни так, чтобы каждый был самостоятельной
   промежуточной предметной областью, переводящей задачу из одного контекста в
   другой. Признак смешения уровней — одно поле, означающее разное в разных
   интерфейсах. Изоляция уровней усложняет интерфейс, но гибкость не бывает
   бесплатной.
3. **Области ответственности.** Сформулируй назначение каждой сущности.
   Модифицирующие операции переформулируй на каждом уровне — они не должны
   «перепрыгивать» уровни. Проверь номенклатуру, встав на место разработчика и
   попробовав написать код по сценариям использования.
4. **Конечные интерфейсы.** Опиши сигнатуры. Правила интерфейса применяй
   осмысленно, а не механически.

Критерии качества API: **очевидность** (как решить задачу), **читаемость**
(правильный код по номенклатуре) и **консистентность** (действие по аналогии).
Большинство проблем рождается не в начальном ядре, а при развитии.
Проектирование «на будущее» небесплатно: избыточные абстракции оправданы, только
если у API это будущее действительно есть, иначе это оверинжиниринг.

Правила конечных интерфейсов: имя сущности должно делать очевидными её действие
и побочные эффекты (за чтением нет модификаций); не экономь на именах; делай тип
поля ясным из названия; ошибки делай информативными; стремись к идемпотентности
операций; не изобретай собственную безопасность и явно декларируй технические
ограничения.

## Обратная совместимость

Совместимость — временна́я характеристика качества: код потребителей должен
продолжать решать свою задачу со временем. Это отличает разработку API от
разработки ПО вообще.

- **Слом совместимости — налог на потребителей.** Не ломай совместимость, если
  это возможно. По semver мажорная версия эквивалентна обратно несовместимому
  изменению. Срок поддержки версии увязывай с жизненным циклом приложений
  потребителей (золотое правило): переписать код им придётся не раньше, чем
  устареют сами приложения.
- **Минимальная публичная поверхность («ватерлиния айсберга»).** Выставляй
  наружу минимум функциональности; каждое публичное поле и метод — продуктовое
  решение и обязательство. Не легитимизируй серые зоны: либо функциональность
  гарантируется всегда, либо на её существование не намекают. Неявные
  договорённости (например, модель консистентности) фиксируй явно.
- **Расширение через абстрагирование.** Расширяй так, чтобы существующие методы
  становились частным случаем более общего интерфейса. Рассматривай каждую
  сущность как реализацию более общего контракта. Вышестоящие сущности должны
  быть информационными контекстами для нижестоящих — сообщать состояние и давать
  средства его менять, а не предписывать поведение. Эти приёмы узнаются как
  принципы SOLID, дополненные понятием уровней абстракции (см.
  `component-design.md`).
- **Слабая связность.** Сильная связность (попытка модифицировать чужую базовую
  функциональность) схлопывает чистоту интерфейсов; параметризуй взаимодействие,
  чтобы разорвать жёсткие зависимости. Изолируй сторонние зависимости
  программной обвязкой с точками расширения, а не проксируй чужой API «как
  есть». Тестируй именно формальные интерфейсы, включая «самоочевидные»
  инварианты.

Совместимость и версионирование пересекаются с разделами `quality-criteria.md` и
`component-design.md` — не дублируй их, опирайся на них.

## Правила веб-уровня (REST поверх HTTP)

Если API строится как REST поверх HTTP, степень «RESTful» удобно оценивать по
модели зрелости Ричардсона (уровни URI, HTTP, Hypermedia). Ниже — практический
свод правил; модальность сохраняй (обязательно / следует / допустимо).

- **URI и ресурсы.** Слеш `/` — для иерархии, без завершающего слеша; дефисы, а
  не подчёркивания; нижний регистр; без расширений файлов. Архетипы ресурсов:
  существительное в единственном числе для документа, во множественном — для
  коллекции и хранилища, глагол — для контроллера. Имена CRUD-функций в URI не
  используются — операцию задаёт HTTP-метод. Фильтрация и пагинация — через
  query (`pageSize`, `pageStartIndex`); сложную выборку выноси в контроллер с
  телом запроса.
- **HTTP-методы.** GET — получение (безопасный, без тела), HEAD — заголовки, PUT
  — вставка и обновление, POST — создание в коллекции и запуск контроллеров,
  DELETE — удаление, OPTIONS — метаданные взаимодействий. Не туннелируй методы
  через GET/POST.
- **Коды состояния.** Выбирай точный код, а не только 200. 2xx: 200 (успех с
  телом), 201 (создание), 202 (асинхронный старт), 204 (пустое тело). 3xx: 301,
  303, 304, 307; 302 не использовать. 4xx: 400, 401, 403, 404, 405, 406, 409,
  412, 415. 5xx: 500. Не передавай ошибки кодом 200.
- **Метаданные и кэширование.** Обязателен `Content-Type`; в ответы добавляй
  `Last-Modified` и `ETag`; для созданного ресурса — `Location`. Хранилище
  поддерживает условные PUT (`If-Match`, при несовпадении — 412). Поощряй
  кэширование (`Cache-Control` с `max-age`); кастомные заголовки не меняют
  поведение методов. Используй прикладные медиатипы и согласование через
  `Accept`.
- **Представления и гипермедиа.** JSON как формат по умолчанию, корректный и в
  camelCase; без лишних «конвертов». Единообразные формы ссылок, связей и
  ошибок; обязательная ссылка `self`. Минимизируй число точек входа (опирайся на
  docroot); ссылки — это контекстно-зависимое меню действий (HATEOAS).
- **Потребности клиентов.** Версию обозначай не в URI (`v2`), а через схемы
  (версии формы) и ETag (версии состояния); новый URI — только для нового
  концепта. Защита ресурсов — OAuth или решения API management. Частичные ответы
  и встраивание связанных ресурсов — через query (`fields`); кросс-доменный
  доступ — CORS (чтение и запись), JSONP — устаревшая альтернатива только для
  чтения.

Учитывай возраст конкретных рекомендаций (свод правил веб-уровня сформулирован в
`2012` году) и сверяй с современной практикой и спецификациями.

## API как продукт

Когда API публикуется наружу, он становится продуктом со своей аудиторией:
пользователи, бизнес-заказчики и разработчики, чьи интересы не совпадают. Это
влияет на архитектурные решения:

- запас прочности и SLA выше обычного: одна строка кода потребителя резко меняет
  нагрузку, а уязвимость API тиражируется на все приложения потребителей;
- для API недоступен дешёвый MVP — проверка гипотез требует, чтобы потребители
  вложились в код; помогает референсное приложение;
- нужна тестовая среда, симулирующая реальные действия и позволяющая
  автоматизировать тестирование на стороне потребителя;
- публикуй роадмапы (хотя бы условные), особенно при прекращении поддержки
  функциональности.

Документация — лицо продукта и часть его контракта. Формальная спецификация (в
том числе OpenAPI) не заменяет документацию — нужны словесные описания,
лаконичные примеры кода и короткий quick start (разросшийся quick start —
признак недоработки частотных сценариев). Подробнее о подготовке документации
см. навык `ait-writing`.
