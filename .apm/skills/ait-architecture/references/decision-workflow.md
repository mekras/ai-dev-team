# Decision Workflow

Используй этот файл, когда нужно подготовить ADR, выбрать между вариантами,
проверить архитектурное решение или честно оформить неопределённость.

## Типы записей

Разделяй сведения по статусу:

| Тип   | Значение                                                        |
| ----- | --------------------------------------------------------------- |
| `F`   | Подтверждённый факт из источника, кода, наблюдения или проверки |
| `A`   | Рабочее допущение                                               |
| `Q`   | Открытый вопрос                                                 |
| `R`   | Риск                                                            |
| `REQ` | Требование или ограничение к решению                            |
| `ODR` | Организационное решение с архитектурными последствиями          |
| `ADR` | Архитектурное решение                                           |

Решение, мнение или прежний вывод не становятся фактом без отслеживаемого
основания.

## Матрица развилки

Перед существенным ADR опиши развилку или граф состояний:

| Поле                        | Что фиксировать                                                      |
| --------------------------- | -------------------------------------------------------------------- |
| Вопрос                      | Какую архитектурную неопределённость закрывает решение               |
| Варианты или состояния      | Полный список, включая неудобные варианты                            |
| Переходы                    | Для графа: возможные переходы и сложность каждого перехода           |
| Источники                   | Факты, допущения, риски, требования, наблюдения                      |
| Поддержка                   | Какие сведения поддерживают вариант                                  |
| Ограничения                 | Какие сведения ослабляют или ограничивают вариант                    |
| Исключение                  | Какие сведения позволяют исключить вариант                           |
| Остаточная неопределённость | Что не даёт оставить один вариант                                    |
| Статус                      | Исключён, ослаблен, жизнеспособен, сильный кандидат, временный выбор |
| Данные для закрытия         | Какие проверки или ответы нужны                                      |
| Следствие                   | Что запрещено или обязательно делать до закрытия развилки            |

Если варианты описывают не выбор, а развитие продукта, оформи граф состояний:
начальное состояние, целевые состояния, допустимые переходы, стоимость и риск
прямых скачков.

## Статусы вариантов

| Статус                                      | Значение                                                       |
| ------------------------------------------- | -------------------------------------------------------------- |
| `Жизнеспособен`                             | Текущие данные позволяют использовать вариант                  |
| `Сильный кандидат`                          | Вариант лучше других объясняет данные, но требует проверки     |
| `Жизнеспособен только как часть комбинации` | Вариант нужен в составе решения, но не как единственная модель |
| `Ослаблен`                                  | Есть существенные доводы против, но данных для исключения мало |
| `Исключён как основная модель`              | Вариант возможен частно, но не годится как архитектурный якорь |
| `Требует данных`                            | Для оценки не хватает проверяемой информации                   |

## Сложность переходов

| Уровень | Значение                                                                          |
| ------- | --------------------------------------------------------------------------------- |
| `L1`    | Низкая сложность: в основном организационный или процессный переход               |
| `L2`    | Средняя сложность: нужны явные границы, документация или небольшое выделение кода |
| `L3`    | Высокая сложность: нужны контракты, ответственность и переработка зависимостей    |
| `L4`    | Очень высокая сложность: велик риск дорогой переработки или потери совместимости  |
| `L5`    | Нежелательный прямой переход без промежуточных состояний                          |

## Анализ компромиссов

В архитектуре редко есть «лучшее» решение и «передовой опыт»; цель — наименее
худшее сочетание компромиссов для конкретного контекста. Используй повторяемый
процесс из трёх шагов:

1. **Найти запутанные измерения** — что в архитектуре переплетено между собой.
   Опирайся на знание существующей системы; начни с диаграммы статического
   сопряжения (зависимости запуска), отделив его от связей, возникающих только
   внутри рабочего процесса.
2. **Проанализировать связи** — как выявленные части влияют друг на друга;
   отбрось заведомо неосуществимые комбинации и оставь силы, требующие оценки.
3. **Оценить компромиссы** — определи влияние изменения каждой силы на
   взаимозависимые части и зафиксируй выбор.

Приёмы, повышающие качество анализа:

- **Качественный анализ важнее количественного**: две архитектуры обычно
  различаются слишком сильно для честного числового сравнения; строй
  сравнительные таблицы характеристик, а не мнимо точные метрики.
- **Списки MECE** (взаимоисключающие, совместно исчерпывающие): сравнивай
  соизмеримые варианты (не очередь сообщений с сервисной шиной) и покрывай всё
  пространство вариантов без пропусков и наложений.
- **Ловушка «выпадения из контекста»**: обобщённые компромиссы без учёта
  конкретной ситуации часто меняются при добавлении контекста; уравновешивай не
  все, а правильный для задачи набор компромиссов.
- **Моделирование сценариев**: вместо построения целых систем проиграй вероятные
  сценарии предметной области («что, если»), чтобы увидеть реальные компромиссы;
  проектирование итеративно.
- **Избегай «панацеи» и пропаганды**: ни один приём не эффективен за пределами
  своего узкого пространства задач; требуй честной оценки сильных и слабых
  сторон и не выбирай инструмент по энтузиазму.
- **Коммуникация**: своди анализ к нескольким ключевым компромиссам и формулируй
  выбор языком цели заинтересованных лиц (что важнее для них), а не списком
  технических деталей.

Этот процесс согласован с матрицей развилки выше: шаги 1–2 наполняют столбцы
«Источники», «Поддержка», «Ограничения», шаг 3 — «Остаточная неопределённость» и
«Следствие». Основание — источник `SAHP`
(`knowledge/data/sahp/items/source/legacy-statements.md`).

## Вопросы для ADR

Каждый существенный ADR должен отвечать:

1. Какую проблему жизненного цикла снижает решение: разработка, освоение,
   сопровождение, развёртывание, обновление, интеграция или эксплуатация?
2. На какую концепцию, требования, предметную область, факты, допущения, риски и
   ограничения оно опирается?
3. Какие варианты рассмотрены и чем они отличаются?
4. Какие данные исключили неподходящие варианты?
5. Если решение временное, почему его нужно принять сейчас?
6. Какую форму системы оно задаёт: элементы, границы, зависимости,
   взаимодействия, режимы развёртывания?
7. Почему выбранный компромисс лучше альтернатив при текущих ограничениях?
8. Какие варианты решение сохраняет открытыми?
9. Какие варианты оно закрывает или делает дороже?
10. Как проверить пользу решения на ближайшем срезе разработки?
11. Кто владеет элементами решения после внедрения?
12. Поддерживает ли структура команды выбранные границы и контракты?
13. Какая ось изменения лежит в основе границы?
14. Где находится авторитетное представление ключевых знаний?
15. Какие виды сопряжения создаёт решение и чем они оправданы?
16. Какие сценарии проверяют решение?
17. Какие границы предметной модели или ограниченные контексты затрагиваются?
18. Почему выбранный стиль или режим разделения уместен сейчас?
19. Какую детальную и динамическую сложность решение добавляет или убирает?
20. Какие петли обратной связи, задержки, побочные эффекты и точки рычага нужно
    учесть?
21. Какие представления нужны, чтобы объяснить решение участникам?
22. При каких новых данных решение нужно пересмотреть?

Если решение опирается на непроверенное требование, сначала уточни статус:
бизнес-требование, пользовательское требование, функциональное требование,
атрибут качества, внешнее интерфейсное требование, ограничение или
предположение. Архитектурное решение не должно скрывать разрыв в требованиях.

Перед фиксацией ADR пройди цепочку «для чего?» для выбранного решения и для
главной отвергнутой альтернативы. Если выбранное решение не даёт более ясного
пути к цели, риску, стоимости или качеству системы, чем альтернатива, решение
ещё не обосновано.

## Архитектурная значимость решения

Не каждое решение требует ADR. Перед оформлением применяй тест архитектурной
значимости: решение значимо, если отвечает на архитектурно значимое требование
(ASR) — измеримо влияет на структуру системы, её качества, стоимость изменения
или сопряжение. Незначимое решение фиксируй короче (заметка в задаче) или не
фиксируй вовсе. Основание — источник `ADRG` (`adr.github.io`).

## Готовность и завершённость решения

Перед фиксацией проверь два среза:

- готовность (Definition of Ready): проблема и контекст ясны, варианты собраны,
  критерии сравнения определены;
- завершённость (Definition of Done): есть доказательства, критерии и
  альтернативы, согласие участников, документирование и план реализации или
  пересмотра.

Основание — `ADRG`.

## Объём и вид ADR

- Один ADR — одно решение. Не объединяй несвязанные решения в одну запись.
- Название ADR — короткое именное словосочетание, отражающее суть решения, а не
  порядковый номер.
- Различай ADR-политику и ADR-случай: ADR-политика задаёт переиспользуемое
  правило (например, «таблицей владеет сервис, который в неё пишет»), ADR-случай
  применяет правило к конкретной ситуации и ссылается на политику.

Основание — анализ ADR источника `SAHP`.

## Выбор шаблона ADR

У проекта нет единого обязательного шаблона ADR. Для каждого решения подбирай
подходящий шаблон по весу решения и согласуй выбор с пользователем до
оформления. Распространённые варианты — Nygard (шаблон Найгарда, по умолчанию),
MADR (когда важен анализ вариантов), Y-Statement (мелкие решения), ISO/IEC/IEEE
42010 (формальные контексты). Перечень не исчерпывающий.

Структуру и готовые скелеты шаблонов см. в `references/adr-templates.md`.

Какой бы шаблон ни был выбран, он должен покрывать минимум: контекст и проблему,
рассмотренные варианты, решение, последствия и способ проверки.

## Заполнение секций

- Контекст: задача, ограничения и рассмотренные альтернативы — без выбранного
  решения. Если альтернатив не видно, решение, скорее всего, ещё не проработано.
- Решение: выбор и обоснование, включая разбор, почему отвергнутые варианты хуже
  при текущих ограничениях. Обоснование не подразумевается, а пишется.
- Последствия: явно называй издержки и принятый компромисс (стоимость, снижение
  гибкости, риск производительности и подобное), а не только плюсы.
- Если решение несёт бизнес-риск или затрагивает другие стороны, фиксируй, кто и
  какой компромисс согласовал.

Основание — анализ ADR источника `SAHP`.

## Статусы ADR

Жизненный цикл решения отражай статусом (источник `ADRG`):

- `Предложено` — решение оформлено, но ещё не принято;
- `Принято` — решение действует;
- `Устарело` — решение больше не рекомендуется, замена не назначена;
- `Заменено` — решение заменено другим ADR (укажи ссылку на заменяющий).

## Ведение ADR

- Указывай условие пересмотра: при каких новых данных или изменениях внешних
  факторов решение нужно пересмотреть. Это обязательная часть, а не опция.
- Если решение требует постоянного контроля соблюдения, свяжи его с функцией
  пригодности (см. `quality-criteria.md`), а не полагайся на ручной надзор;
  реализацию сформулированной функции как исполняемой проверки передавай в навык
  `ait-code-testing` (цепочка «решение → функция пригодности → проверка»).
- Веди реестр ADR (индекс со ссылками на решения и затронутые области), чтобы
  решения были прослеживаемы и не терялись.

Основание — анализ ADR источника `SAHP`.

Если документ нужен для команды, а не для истории решений, вместо ADR может
хватить короткой архитектурной заметки в задаче.
