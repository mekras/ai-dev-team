# DHARMA: доменная иерархия доступа для микросервисов

Эта справка описывает модель управления доступом к API в микросервисной
архитектуре. Основание — корпус знаний проекта (`knowledge/statements/SMAS.md`,
отчёт `Securing Microservice APIs`). Модель платформонезависима: её можно
реализовать на разных платформах.

## Базовые понятия

- **Трастовый домен** — множество сервисов, взаимодействующих привилегированно.
- **Доменное отношение** — причина группировки сервисов в домен (общая
  платформа, сетевой сегмент, бизнес-близость).
- **Механизм доверия** — способ, которым сервисы внутри домена проверяют, что
  запрос пришёл из доверенного источника.
- **Механизм доступа** — способ аутентификации и авторизации запросов,
  приходящих в домен извне.
- **Внутренний (interior) эндпойнт** — доступен сервисам внутри домена через
  механизм доверия.
- **Граничный (boundary) эндпойнт** — доступен внешним сервисам через механизм
  доступа.
- **Доменная иерархия** — граничный эндпойнт одного домена может быть внутренним
  для внешнего домена, образуя вложенность доменов.

Приём оптимизации: проверка доверия в рантайме обычно дешевле аутентификации
недоверенного источника, поэтому различие «доверенный / недоверенный запрос»
используется для оптимизации решения.

## Методология проектирования (четыре шага)

1. Выявить трастовые домены и их отношения; продумать вложенность доменов.
2. Определить механизм доверия и механизм доступа для каждого домена.
3. Определить внутренние и граничные эндпойнты сервисов.
4. Выбрать платформы и компоненты реализации эндпойнтов (часто граничные
   эндпойнты реализуются на посредниках — шлюзах и сервисных прокси).

## Платформонезависимая реализация в три уровня

- **Внешняя область** (вне контроля организации): доступ к внешнему домену —
  OAuth 2.0 с непрозрачным токеном, который нельзя вывести извне.
- **Внешний домен** (грубозернистые переиспользуемые сервисы): доверие на основе
  сертификатов и подписанных JWT; это же — механизм доступа внутренних доменов.
- **Внутренние домены** (самые гранулярные сервисы): доверие смещено к
  производительности (изоляция сети, коллокация), но JWT может передаваться для
  учёта и наблюдаемости.

## Соображения реализации

- **Управление сертификатами**: удостоверяющий центр выдаёт сертификаты (как
  минимум по одному на внутренний домен).
- **Управление токенами**: валидация, выпуск, обмен и разыменование токенов;
  желательно распределённые защищённые токен-сервисы для снижения задержки.
- **Короткоживущие токены**: внутри доменов ожидаются JWT со сроком жизни менее
  часа, возможно одноразовые; области OAuth (scopes) и утверждения JWT (claims)
  несут информацию для авторизации.
- **Безопасное снабжение** компонентов сертификатами и аудит всех операций с
  токенами и решений об авторизации (обычно на посредниках).

## Границы применимости

Отчёт-первоисточник ограничен HTTP-API и не охватывает не-HTTP-протоколы и
безопасность контейнеров в целом. Конкретные технологии и продукты отражают
состояние экосистемы на 2018 год; при применении сверяйся с актуальными
материалами. Модель DHARMA — основание для проектирования, а не принятое решение
проекта: уровень принимаемого риска утверждает человек.
