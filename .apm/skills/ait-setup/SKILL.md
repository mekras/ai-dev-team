# Навык: настройка проекта

## Назначение

Навык помогает настроить новый проект или перенастроить существующий проект под
`ai-dev-team` через короткий диалог и минимальные безопасные изменения.

Главная роль `ait-setup` — не повторять процедуры соседних навыков, а задать
обязательный порядок проверки, маршрутизации и вопросов к человеку.

## Когда применять

Применяй навык, если пользователь просит:

- настроить проект;
- перенастроить существующий проект;
- обновить проект под актуальные правила `ai-dev-team`;
- перейти на новую версию `ai-dev-team` и пересмотреть проект под её изменения;
- начать новый проект;
- инициализировать проект;
- подготовить стартовые файлы проекта;
- завести базовый `README.md`, `LICENSE`, `AGENTS.md`, `.gitignore` и при
  необходимости `CHANGELOG.md`;
- подготовить новый репозиторий к работе с агентами.

Не применяй навык для обычной правки одного существующего файла, настройки
только подагентов, создания одной лицензии или полного проектирования
документации.

## Режим по умолчанию

По умолчанию работай в диалоговом режиме: задавай по одному обязательному
вопросу за ход и не создавай окончательные файлы, пока не собраны блокирующие
решения человека.

Диалоговый режим означает, что агент не принимает проектные решения молча. Перед
каждым решением, которое меняет маршрут настройки, структуру проекта, статус
корпуса знаний, лицензию, клиентские инструкции или файлы проекта, сформулируй
предлагаемое решение и получи подтверждение человека. Перед каждой записью в
файл назови файл, смысл изменения и причину, затем дождись подтверждения.

## Обязательный порядок

1. Сначала примени `ai-audit-project`: проверь текущие стартовые файлы, правила
   проекта для агента и явные расхождения с актуальными правилами продукта.
2. Если текущая настройка относится к публикуемой коллекции APM, обязательно
   примени `ai-setup-apm` и не дублируй его проверку структуры, `apm.yml` и
   процесса `apm run tests` внутри `ait-setup`. Определяй этот маршрут по
   назначению проекта и публикуемым примитивам APM, а не по одному найденному
   файлу.
3. Если задача — переход на новую версию `ai-dev-team` (человек просит обновить
   проект под новую версию либо аудит показал смену версии продукта), проведи
   пересмотр по `references/setup-dialogue.md`, раздел «Обновление версии
   продукта», прежде чем продолжать остальные шаги настройки.
4. Если в корне проекта уже есть `.ai-dev-team-state.yml`, сначала прочитай его
   и используй как источник ранее принятых параметров настройки проекта:
   видимости, Git, журнала изменений, корпуса знаний и состояния клиентских
   инструкций. Не дублируй эти сведения в `AGENTS.md`, если они нужны только
   самому `ait-setup`.
5. Собери недостающие сведения по `references/setup-dialogue.md`: один
   блокирующий вопрос за ход, начиная с самого раннего решения, без которого
   нельзя безопасно продолжить.
6. Если проект подключён к `ai-dev-team` через APM как потребитель, настрой
   клиентские инструкции по правилам `references/setup-dialogue.md`: сначала
   согласуй сохранение рукописных локальных правил из `AGENTS.md` или
   `CLAUDE.md` в локальном слое проекта. Обновляй клиентские инструкции через
   реальную компиляцию только для уже сгенерированных файлов или после
   отдельного явного решения человека заменить рукописный файл сгенерированным.
   Не выноси `apm compile --dry-run` в обычный следующий шаг, если человек не
   просит диагностировать генерацию или риск перезаписи.
7. Для `README.md` применяй `ait-readme`.
8. Для `AGENTS.md` применяй `ai-agents-md-maintenance`, но держи его коротким
   машинным входом: локальные рабочие правила, проектные границы и ссылки на
   нестандартные источники. Параметры настройки проекта, которые нужны только
   `ait-setup`, сохраняй в `.ai-dev-team-state.yml`.
9. Для журнала изменений:
   - если в проекте уже есть `CHANGELOG.md`, `NEWS.md` или `HISTORY.md`,
     предложи закрепить правило его сопровождения через `ait-changelog`;
   - если журнала нет и в `.ai-dev-team-state.yml` не зафиксирован явный отказ,
     предложи создать `CHANGELOG.md` как дополнительный стартовый файл;
   - если пользователь отказался от журнала изменений, зафиксируй это решение в
     `.ai-dev-team-state.yml`, чтобы не предлагать журнал повторно без нового
     запроса человека.
10. Для корпуса знаний:

- если корпус уже есть, применяй `kc-setup`;
- если корпуса нет, предложи создать его через `kc-setup`;
- если пользователь отказался от корпуса, зафиксируй это решение в
  `.ai-dev-team-state.yml`, чтобы не задавать тот же вопрос повторно без нового
  запроса человека.

11. Если настройка показывает, что для осмысленного `README.md` не хватает
    концепции проекта, сначала согласуй это с человеком и затем применяй
    `ait-docs-concept`.
12. Если проект использует Git, создавай или обновляй `.gitignore` минимально
    под известный стек и общие локальные артефакты. Если стек неизвестен,
    используй только универсальные безопасные исключения.
13. Создавай `LICENSE` только на основании выбранной лицензии или явного решения
    человека. Не подставляй текст лицензии по умолчанию.
14. После стартовых файлов предложи настройку подагентов через
    `ai-setup-subagents`.
15. Если пользователь хочет приватный локальный слой знаний или неотслеживаемые
    локальные правила, применяй `ait-private-knowledge`.
16. Перед ответом «настройка закончена» проверь критерий завершения из
    `references/setup-dialogue.md`: по каждому обязательному направлению должен
    быть статус «сделано», «не требуется», «отложено решением человека» или
    «заблокировано вопросом». Если есть заблокированный пункт, настройка не
    закончена.
17. Перед завершением проверь, какие файлы созданы или изменены, и добавь новые
    отслеживаемые файлы в индекс Git, если проект использует Git и правила
    проекта не запрещают индексацию.

## Что читать дополнительно

- `references/setup-dialogue.md` — порядок вопросов, карта маршрутов по навыкам
  и правила для корпуса знаний.
- `assets/minimal-files.md` — рабочая заготовка минимального стартового набора.

## Границы

- Не дублируй процедуры `ai-audit-project`, `ai-setup-apm`, `ait-readme`,
  `ai-agents-md-maintenance`, `ait-changelog`, `kc-setup` и `ai-setup-subagents`
  внутри этого навыка.
- Не пропускай стартовый аудит проекта даже для нового запроса «Настрой проект».
- Не маршрутизируй проект в `ai-setup-apm`, пока не определено, что текущая
  задача относится к публикуемой APM-поверхности проекта.
- Не передавай запросы «Обнови проект под новую версию ai-dev-team» и «Обнови
  проект под новую версию» в `ai-setup-apm`, если человек уже обновил
  установленную версию продукта и просит именно пересмотр проекта по журналу
  изменений.
- Не вноси файл, клиентскую инструкцию или проектное решение без отдельного
  подтверждения человека.
- Не считай перенос смысла рукописного `AGENTS.md` или `CLAUDE.md` в
  `.apm/instructions/**` разрешением перезаписать исходный рукописный файл через
  `apm compile`.
- Не поднимай `apm compile --dry-run` как штатный следующий шаг настройки для
  рукописных клиентских инструкций; это только явная диагностика риска
  генерации, когда она нужна для решения человека.
- Не создавай корпус знаний молча и не выбирай имя его папки без решения
  человека.
- Не создавай журнал изменений молча и не выбирай его формат без решения
  человека, если в проекте уже используется другой журнал или есть несколько
  кандидатов.
- Не используй `AGENTS.md` как хранилище ответов на анкету настройки, если эти
  сведения не нужны агенту в обычной предметной работе.
- Не предлагай корпус знаний повторно, если в `.ai-dev-team-state.yml` уже
  зафиксировано решение проекта работать без него.
- Не предлагай журнал изменений повторно, если в `.ai-dev-team-state.yml` уже
  зафиксировано решение проекта работать без него.
- Не добавляй в `AGENTS.md` служебные параметры вроде видимости проекта, статуса
  Git или раздела «Проверки», если они нужны только маршруту `ait-setup` и не
  являются рабочими правилами проекта.
- Не создавай крупную структуру каталогов без отдельного обоснования.
- Не выдумывай концепцию, требования, архитектуру, стек, сроки, роли и
  бизнес-эффект.
- При обновлении версии не придумывай действия, которых нет в разделе «Действия
  при обновлении» журнала изменений продукта, и не вноси массовые правки проекта
  без подтверждения каждого изменения.
- Не записывай настоящие секреты в `AGENTS.md`, `.env`, локальные правила или
  другие текстовые файлы.

## Результат

Результатом работы должны быть:

- минимальные стартовые файлы проекта или вопросы, без которых их нельзя создать
  безопасно;
- статус обязательных направлений настройки: README, правила для агентов,
  клиентские инструкции, журнал изменений, корпус знаний, `.gitignore`,
  лицензия, подагенты, Git и проверки;
- если менялись или подтверждались параметры настройки проекта, обновлённый
  `.ai-dev-team-state.yml`;
- явный список созданных и изменённых файлов;
- перечень применённых навыков и причин их выбора;
- список решений человека, которые были использованы;
- открытые вопросы и следующие безопасные шаги.

Для перехода на новую версию результат дополнительно содержит применённую версию
из `.ai-dev-team-state.yml`, целевую версию установленного продукта, перечень
рассмотренных действий из журнала изменений и решения человека по каждому из
них.
