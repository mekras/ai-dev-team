# Навык: настройка проекта

## Назначение

Навык помогает настроить новый проект или перенастроить существующий проект под
`ai-dev-team` через короткий диалог и минимальные безопасные изменения.

Главная роль `ait-setup` — не повторять процедуры соседних навыков, а задать
обязательный порядок проверки, маршрутизации и вопросов к человеку.

## Когда применять

Применяй навык, если пользователь просит:

- настроить проект;
- перенастроить существующий проект;
- обновить проект под актуальные правила `ai-dev-team`;
- начать новый проект;
- инициализировать проект;
- подготовить стартовые файлы проекта;
- завести базовый `README.md`, `LICENSE`, `AGENTS.md` и `.gitignore`;
- подготовить новый репозиторий к работе с агентами.

Не применяй навык для обычной правки одного существующего файла, настройки
только подагентов, создания одной лицензии или полного проектирования
документации.

## Режим по умолчанию

По умолчанию работай в диалоговом режиме: задавай по одному обязательному
вопросу за ход и не создавай окончательные файлы, пока не собраны блокирующие
решения человека.

## Обязательный порядок

1. Сначала примени `ai-audit-project`: проверь текущие стартовые файлы, правила
   проекта для агента и явные расхождения с актуальными правилами продукта.
2. Если текущий проект является коллекцией APM, обязательно примени
   `ai-apm-setup` и не дублируй его проверку структуры, `apm.yml` и процесса
   `apm run tests` внутри `ait-setup`.
3. Собери недостающие сведения по `references/setup-dialogue.md`: один
   блокирующий вопрос за ход, начиная с самого раннего решения, без которого
   нельзя безопасно продолжить.
4. Для `README.md` применяй `project-readme`.
5. Для `AGENTS.md` применяй `ai-agents-md-maintenance`.
6. Для корпуса знаний:
   - если корпус уже есть, применяй `kc-setup`;
   - если корпуса нет, предложи создать его через `kc-setup`;
   - если пользователь отказался от корпуса, зафиксируй это решение в
     `AGENTS.md`, чтобы не задавать тот же вопрос повторно без нового запроса
     человека.
7. Если настройка показывает, что для осмысленного `README.md` не хватает
   концепции проекта, сначала согласуй это с человеком и затем применяй
   `ait-docs-concept`.
8. Если проект использует Git, создавай или обновляй `.gitignore` минимально под
   известный стек и общие локальные артефакты. Если стек неизвестен, используй
   только универсальные безопасные исключения.
9. Создавай `LICENSE` только на основании выбранной лицензии или явного решения
   человека. Не подставляй текст лицензии по умолчанию.
10. После стартовых файлов предложи настройку подагентов через
    `ai-subagent-setup`.
11. Если пользователь хочет приватный локальный слой знаний или неотслеживаемые
    локальные правила, применяй `private-project-knowledge`.
12. Перед завершением проверь, какие файлы созданы или изменены, и добавь новые
    отслеживаемые файлы в индекс Git, если проект использует Git и правила
    проекта не запрещают индексацию.

## Что читать дополнительно

- `references/setup-dialogue.md` — порядок вопросов, карта маршрутов по навыкам
  и правила для корпуса знаний.
- `assets/minimal-files.md` — рабочая заготовка минимального стартового набора.

## Границы

- Не дублируй процедуры `ai-audit-project`, `ai-apm-setup`, `project-readme`,
  `ai-agents-md-maintenance`, `kc-setup` и `ai-subagent-setup` внутри этого
  навыка.
- Не пропускай стартовый аудит проекта даже для нового запроса «Настрой проект».
- Не создавай корпус знаний молча и не выбирай имя его папки без решения
  человека.
- Не предлагай корпус знаний повторно, если в `AGENTS.md` уже зафиксировано
  решение проекта работать без него.
- Не создавай крупную структуру каталогов без отдельного обоснования.
- Не выдумывай концепцию, требования, архитектуру, стек, сроки, роли и
  бизнес-эффект.
- Не записывай настоящие секреты в `AGENTS.md`, `.env`, локальные правила или
  другие текстовые файлы.

## Результат

Результатом работы должны быть:

- минимальные стартовые файлы проекта или вопросы, без которых их нельзя создать
  безопасно;
- явный список созданных и изменённых файлов;
- перечень применённых навыков и причин их выбора;
- список решений человека, которые были использованы;
- открытые вопросы и следующие безопасные шаги.
