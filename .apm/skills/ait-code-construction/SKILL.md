# ait-code-construction

Помогай конструировать код как дисциплину, а не как разовое написание строк:
детальное проектирование на уровне реализации, классы, методы, данные,
управляющие структуры, защитное программирование и постепенное улучшение
работающего кода.

Навык даёт порядок конструирования и связывает его с готовыми решениями
архитектуры и требований. Он не выбирает архитектурные границы и не проверяет
готовый код вместо обзора — для этого есть соседние навыки.

## Когда применять

Навык применяется, если нужно:

- спроектировать класс, метод или структуру данных на уровне реализации;
- выбрать управляющие структуры так, чтобы порядок выполнения был понятен;
- применить защитное программирование к входным данным, ошибкам и исключениям;
- улучшить читаемость кода: имена, функции, форматирование, комментарии;
- выполнить пошаговый рефакторинг работающего кода без потери поведения;
- спланировать интеграцию изменений в систему с проверкой на каждом шаге.

## Когда не применять

- нужно выбрать архитектурные границы, направление зависимостей или режим
  разделения системы — используй `ait-architecture`;
- нужно проверить уже готовый код на соответствие задаче и критериям — используй
  `ait-code-review`;
- нужно спроектировать или дополнить набор тестов разработчика — используй
  `ait-code-testing`;
- задача является изменением требований, а не их реализацией.

## Порядок работы

1. Проверь готовность оснований: определение проблемы, требования, архитектура и
   выбранные методики разработки достаточно ясны для конструирования (CCMP).
2. Спроектируй решение на уровне реализации эвристически: сочетай абстракцию,
   сокрытие информации, простоту, высокую связность и низкое сцепление (CCMP).
3. Для класса проверь ясность абстракции, качество интерфейса, распределение
   обязанностей и наличие разумной причины для выделения класса; держи класс
   небольшим по ответственности с понятной причиной для изменения (CCMP, CLCD).
4. Для метода удержи одну понятную цель, ограниченный объём, один уровень
   абстракции, содержательное имя и аккуратную работу с параметрами; ограничивай
   число аргументов и избегай аргументов-флагов и смешения команд с запросами
   (CCMP, CLCD). При сложном алгоритме отделяй намерение от синтаксиса через
   псевдокод и пошаговую разработку (CCMP).
5. Для данных дай переменной единственную цель, ограниченную область видимости,
   предсказуемую инициализацию и имя, отражающее смысл; различай объект,
   скрывающий данные за поведением, и структуру данных, открывающую данные
   (CCMP, CLCD).
6. Выбирай управляющие структуры так, чтобы порядок выполнения читался; глубокую
   вложенность и нестандартные переходы обосновывай отдельно (CCMP).
7. Применяй защитное программирование: явно обрабатывай неправильные входные
   данные, ошибки и исключения, сохраняй читаемым основной путь выполнения,
   передавай контекст исключения и избегай возврата и передачи `null`, не
   превращая защиту в самостоятельный источник сложности (CCMP, CLCD).
8. Изолируй границы со сторонним кодом, чтобы детали чужого API не
   распространялись по системе (CLCD, CLAR).
9. Поддерживай читаемость: имена передают намерение и удобны для поиска,
   форматирование держит связанные элементы рядом и делает структуру файла
   предсказуемой, комментарий объясняет намерение, а не компенсирует плохой код
   (CLCD, CCMP).
10. Рефакторинг и очищение веди маленькими проверяемыми шагами: каждое улучшение
    сохраняет работоспособность, а вид улучшения осознан; оставляй код чище, чем
    он был (CCMP, CLCD).
11. Интегрируй изменения по осознанной стратегии и достаточно часто, чтобы не
    откладывать обнаружение дорогих дефектов (CCMP).

## Границы

- Не выбирай архитектурные границы и направление зависимостей вместо
  `ait-architecture`; опирайся на уже принятые архитектурные решения.
- Не используй запахи кода и эвристики как механическую замену инженерного
  суждения (CLCD).
- Не переноси крупные дословные примеры кода из источников `CCMP` и `CLCD`:
  лицензия ограничена, перенос идей выполняй своими словами.
- Не выдавай локальное предположение о реализации за принятое решение, если
  выбор должен сделать владелец.

## Вывод

Сообщай: что сконструировано или изменено, какие решения приняты на уровне
классов, методов, данных и управляющих структур, как обработаны ошибки и
границы, какими шагами шёл рефакторинг и как изменения интегрированы.

## Опорные источники

Навык опирается на источники корпуса: сначала найди источник по `id` в
`knowledge/catalog.yml`, затем используй `knowledge/data/<slug>/source.yml`,
`knowledge/data/<slug>/items.yml` и связанные утверждения: `CCMP`, `CLCD`,
`CLAR`.
