# README коллекции APM с навыками и правилами

Применяй эту справку только для главного `README.md` проекта, который поставляет
через APM навыки, правила или инструкции для агентов.

## Определение сценария

Сначала проверь, кто получает пользу от коллекции после установки. Не назначай
сопровождающего репозитория основной аудиторией только потому, что в проекте
есть файл блокировки, сгенерированные инструкции или команды сборки.

Пользовательский сценарий должен объяснять:

- зачем устанавливать коллекцию;
- какой командой подключить её к проекту или среде пользователя;
- что станет доступно агенту после установки;
- какое первое практическое действие выполнить после установки.

Локальное сопровождение репозитория должно быть отдельным разделом. Команды
`apm install --frozen`, `apm compile`, `apm run compile`, `git status --short` и
похожие команды из корня репозитория относятся к сопровождению, пока не
доказано, что это первый пользовательский сценарий продукта.

## Заголовок и описание

Если имя репозитория является slug, а не собственным названием, выбери
человекочитаемый заголовок по назначению коллекции: например, «Навыки для
OpenClaw» вместо `openclaw-skills`.

Первый экран README должен описывать назначение коллекции и пользу для
пользователя, а не состав репозитория, зависимость на общий пакет, файл
блокировки или сгенерированные инструкции.

## Язык

Для русской прозы применяй `ru-lang` и перестраивай технические гибриды:

- `APM-пакет` → `пакет APM`;
- `APM-проект` → `проект APM`;
- `OpenClaw skills` → `навыки OpenClaw`;
- `lockfile` → `файл блокировки`.

## Цели клиентов

Не делай Codex, Claude или другую цель клиента единственной точкой входа в
README, если проект не ограничен этим клиентом и пользователь явно не выбрал
его. Упоминания `codex`, `claude`, `--target codex`, `--target claude` и похожих
целей должны соответствовать реальной аудитории и области действия команды.

## Ошибочное поведение

README коллекции APM не должен:

- начинаться со структуры репозитория;
- выдавать список каталогов за объяснение пользы проекта;
- помещать команды сопровождения в пользовательский быстрый старт;
- использовать локальный путь как основной способ установки для внешнего
  пользователя;
- описывать файл блокировки, зависимости, сгенерированные инструкции и рабочее
  дерево раньше назначения коллекции и пользовательского применения.
