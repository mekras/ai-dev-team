# Навык: сопровождение журнала изменений

## Назначение

Навык задаёт порядок создания, обновления и проверки журнала изменений проекта.

Журнал изменений нужен для людей, которые сопровождают, используют или обновляют
проект. Он не должен превращаться в ленту коммитов, журнал решений или пересказ
внутренней реализации.

## Когда применять навык

Навык должен применяться, если нужно:

- создать новый `CHANGELOG.md`;
- обновить существующий `CHANGELOG.md`, `NEWS.md` или `HISTORY.md`;
- проверить, нужно ли отразить изменение в журнале;
- добавить действия при обновлении после изменения контракта проекта.

Навык не должен применяться для:

- журнала изменений самого продукта `ai-dev-team`, если задача относится к
  выпуску этого репозитория;
- правки `README.md`, `AGENTS.md`, требований или архитектурной документации;
- локальной правки кода без внешне значимого эффекта.

## Входные данные

Для применения навыка должны быть доступны:

- описание или diff реального изменения;
- понимание, для кого это изменение заметно;
- существующий файл журнала, если он уже есть;
- сведения о действиях при обновлении, если изменение затрагивает контракты,
  форматы, правила или структуру проекта.

Если этих данных недостаточно, их нужно сначала собрать или запросить.

## Что читать дополнительно

- `assets/changelog-template.md` — минимальная заготовка нового `CHANGELOG.md`.

## Порядок применения

### 1. Найти журнал проекта и определить его формат

Сначала проверь, использует ли проект один из файлов:

- `CHANGELOG.md`;
- `NEWS.md`;
- `HISTORY.md`.

Если файл уже есть, сохрани его формат, заголовки, язык и способ записи версий.
Не переписывай существующий журнал под другой стиль только потому, что тебе
нравится другой шаблон.

Если в проекте есть несколько кандидатов, не выбирай между ними молча: покажи
человеку найденные варианты и попроси подтвердить основной журнал.

### 2. Определить, заслуживает ли изменение записи

В журнал изменений попадают внешне значимые изменения:

- пользовательское поведение;
- публичный API, CLI, конфигурация или формат данных;
- требования к обновлению, миграции и ручным действиям;
- заметные исправления дефектов;
- добавление, удаление или переименование возможностей, на которые опирается
  пользователь или сопровождающий.

Для небольших изменений документации выбирай самую короткую правдивую
формулировку по новому ориентиру для читателя: новый справочник, новая
инструкция, новый раздел, исправленная неточность. Не перечисляй переносы,
сокращения, перестановки текста и другие редакторские подробности, если они не
создают отдельного последствия для пользователя или сопровождающего.

Не записывай в журнал:

- чистое форматирование;
- внутренний рефакторинг без внешнего эффекта;
- временную диагностику;
- служебную перестановку файлов без изменения поведения;
- коммиты вида "обновлён lockfile", если пользователь проекта ничего не должен
  знать или делать.

Если значимость изменения неочевидна, вынеси это как вопрос человеку.

### 3. Отделить внешний эффект от внутренней механики

Перед записью ответь:

- что именно изменилось для пользователя, сопровождающего или интегратора;
- как это заметить;
- нужно ли что-то сделать при обновлении;
- что из реализации не нужно в журнале.

Формулируй запись по наблюдаемому результату, а не по внутренним шагам
реализации.

### 4. Выбрать место записи

Если в журнале есть раздел `Невыпущено` или `Unreleased`, добавляй запись туда,
если задача не просит оформить уже выпущенную версию.

Если проект использует версионные разделы без `Невыпущено`, следуй существующей
структуре и не добавляй новый раздел без подтверждённого решения.

Если создаётся новый журнал, используй шаблон из `assets/changelog-template.md`
только как стартовую точку. Не добавляй пустые разделы, которые проект пока не
использует.

### 5. Добавить действия при обновлении, когда они действительно нужны

Если изменение требует действий в проекте-потребителе или у пользователя, добавь
отдельный подраздел `Действия при обновлении` или его существующий эквивалент.

В этот раздел попадают только реальные действия:

- переименование или удаление примитива;
- изменение формата конфигурации или данных;
- изменение структуры проекта;
- шаги миграции;
- ручные действия после обновления.

Не добавляй этот подраздел, если обновление не требует действий.

### 6. Проверить запись перед завершением

После правки перечитай журнал и проверь:

- сохранён ли существующий формат проекта;
- названы ли внешне значимые изменения, а не внутренняя механика;
- нет ли ложных заявлений о поведении, которого не было;
- не придуманы ли прошлые версии, старые записи или действия при обновлении;
- не превратился ли журнал в пересказ коммита или diff.

## Результат применения навыка

Результатом должен быть журнал изменений, который:

- сохраняет формат проекта;
- отражает только внешне значимые изменения;
- помогает понять последствия обновления;
- не требует читать diff или историю Git для базового понимания.

## Что считается ошибкой применения навыка

- переписан существующий формат журнала без решения человека;
- в журнал попала внутренняя механика вместо внешнего эффекта;
- запись добавлена для незначимого технического изменения;
- придуманы действия при обновлении, которых проект не требует;
- задним числом переписана история проекта;
- журнал выбран молча, хотя в проекте было несколько кандидатов.

## Связь с другими элементами продукта

Этот навык должен применяться вместе:

- с ролью технического писателя;
- с `ait-writing` как базовым навыком точного проектного письма;
- с `ait-setup`, если журнал предлагается или создаётся при настройке проекта;
- с `ai-agents-md-maintenance`, если правило ведения журнала закрепляется в
  `AGENTS.md`.
